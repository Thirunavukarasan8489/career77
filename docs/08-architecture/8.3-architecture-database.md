1. Purpose

This document defines the database architecture for Career77.

Career77 uses MongoDB as the primary database and Mongoose as the ODM.

The database stores:

Users
Candidate profiles
Recruiters
Companies
Jobs
Applications
Interviews
Training
Enrollments
Payments
Subscriptions
Notifications
Support tickets
Audit logs
2. Database Technology
MongoDB
    ↓
Mongoose ODM

MongoDB is responsible for persistent application data.

Mongoose provides:

Schemas
Models
Validation
Indexes
Query support
Middleware
3. Database Architecture
Next.js Application
    ↓
Service Layer
    ↓
Mongoose Models
    ↓
MongoDB

Application code should access MongoDB through a centralized database connection and Mongoose models.

4. Database Connection

The MongoDB connection should be managed centrally.

Recommended:

lib/
└── db/
    └── mongodb.ts

The application should reuse existing connections where possible.

Avoid creating a new database connection for every request.

5. Environment Variable

The MongoDB connection string should be stored in an environment variable.

MONGODB_URI

Example:

MONGODB_URI=mongodb+srv://...

The actual value must never be committed to source control.

27. Reference Strategy

MongoDB references should be used for major entities.

Example:

candidateId
jobId
companyId
recruiterId
applicationId
trainingId
paymentId

Use references when data is independently managed or can grow significantly.

28. Embedding Strategy

Embedding may be appropriate for small data that belongs tightly to a parent document.

Examples:

Job
    └── Benefits

Candidate Profile
    └── Skills

Company
    └── Social Links

Avoid embedding large or frequently growing datasets.

29. Data Validation

Mongoose schemas should define validation rules.

Validate:

Required Fields
Email Format
Enum Values
String Length
Number Ranges
ObjectId References

Server-side validation is mandatory.

31. Unique Indexes

Unique indexes may be used where duplicate data is not allowed.

Examples:

users.email
companies.slug
jobs.slug

For applications:

candidateId + jobId

may be used as a compound unique index if duplicate applications are not allowed.

The migration should be considered based on actual product requirements and usage.

37. Data Ownership

Database queries must respect resource ownership.

Examples:

Candidate
    ↓
Own Profile
Own Applications
Own Training Enrollments

Recruiter
    ↓
Authorized Company
Authorized Jobs
Authorized Applicants

Super Admin
    ↓
Platform Data

Ownership checks should be performed before modifying protected data.

38. Soft Delete

Soft deletion may be used for important business records.

Instead of immediately deleting:

deletedAt

or:

isDeleted

may be used.

This can help preserve historical data and auditability.

39. Data Retention

Important business records should not be permanently deleted without considering:

Audit requirements
Legal requirements
Business requirements
Data recovery

Retention rules should be defined as the platform matures.

40. Transactions

MongoDB transactions should be considered when multiple related database operations must succeed together.

Example:

Payment Verified
    ↓
Payment Record Created
    ↓
Training Enrollment Created

If transactional consistency is required, these operations should be handled appropriately.

41. Payment Data Integrity

Payment verification must be server-side.

The database should not grant training access solely because the frontend reports successful payment.

Flow:

Payment Provider
    ↓
Server Verification
    ↓
Payment Record
    ↓
Enrollment
    ↓
Access Granted
42. Recruiter Verification Data Integrity

Recruiter verification should maintain clear status transitions.

Example:

pending
    ↓
verified

or:

pending
    ↓
rejected
    ↓
review_requested
    ↓
verified

Important verification actions should be recorded in audit logs.

43. Database Security

Database security should include:

Strong authentication
Restricted database access
Environment-based credentials
Network restrictions where appropriate
Encrypted connections
Regular backups
Least-privilege access

Database credentials must never be exposed to the client.

44. Backup and Recovery

Production database backups should be configured.

Backup strategy should support:

Regular Backups
    ↓
Backup Retention
    ↓
Recovery Testing

The exact backup schedule should be defined according to production requirements.

45. Database Monitoring

Monitor:

Query performance
Slow queries
Database connections
Storage usage

Performance issues should be investigated before adding unnecessary complexity.

46. Development Environment

Development should use a separate database or database environment.

Recommended:

Development Database
    ↓

When changing:

Data types
Required fields
Relationships

Migration scripts may be required for major changes.
