1. Purpose

This document defines the file storage architecture for Career77.

Career77 uses Cloudinary for storing and managing uploaded files and media.

Primary file types include:

Candidate resumes
Candidate certificates
Profile images
Company logos
Training images
Other approved platform media
2. File Storage Technology

Career77 uses:

Next.js
    ↓
Upload API
    ↓
Cloudinary
    ↓
File URL
    ↓
MongoDB

MongoDB stores file metadata and references.

The actual file content is stored in Cloudinary.

Update Own Resume
Delete Own Resume

Super Admin:

Administrative Access

13. Cloudinary Folder Organization

Cloudinary folders should follow a clear structure.

Example:

career77/
├── profiles/
├── resumes/
├── certificates/
├── companies/
├── trainings/
└── system/

Additional folders may be introduced as the platform grows.

14. File Naming

File names should be generated or normalized by the server.

Avoid relying on user-provided file names for storage identifiers.

Example:

career77/resumes/{userId}/{uniqueFileId}

This helps avoid file name collisions.

15. File Metadata

MongoDB should store only required file metadata.

Example:

{
  "url": "...",
  "publicId": "...",
  "fileName": "...",
  "fileType": "...",
  "fileSize": 123456,
  "uploadedAt": "..."
}

The actual file should remain in Cloudinary.

16. Public and Private Files

Files should be categorized based on access requirements.

Public

Examples:

Company Logo
Public Profile Image
Training Thumbnail
Restricted

Examples:

Resume
Certificate
Private Documents

Access should be controlled according to the resource and user permissions.

17. Resume Privacy

Candidate resumes contain personal information.

Therefore:

Resume
    ↓
Private / Restricted Access

A recruiter should only access a candidate's resume when permitted by the hiring workflow.

21. Replace File

When replacing a file:

New File
    ↓
Validate
    ↓
Upload New File
    ↓
Update Database Reference
    ↓
Delete Old File

The old file should be deleted only after the new file has been successfully uploaded and saved.

22. Failed Upload Handling

If Cloudinary upload fails:

Upload Failed
    ↓
Do Not Save Invalid Reference
    ↓
Return Error

If the database update fails after a successful Cloudinary upload, the system should handle the orphaned file appropriately.

23. Orphaned Files

Orphaned files may occur when:

Cloudinary Upload
    ↓
Database Save Failed

The system should have a cleanup strategy for unused files.

24. File Validation

Validation should include:

Allowed MIME Type
Allowed Extension
Maximum File Size
File Content

Reject unsupported files.

Example:

.exe
.sh
.bat

should not be accepted for normal document uploads.

25. File Security

Career77 should:

Validate uploads server-side
Restrict file types
Limit file sizes
Avoid exposing unnecessary file metadata
Protect private files
Use secure Cloudinary configuration
Prevent unauthorized deletion
26. Environment Variables

Cloudinary credentials should be stored in environment variables.

Example:

CLOUDINARY_CLOUD_NAME
CLOUDINARY_API_KEY
CLOUDINARY_API_SECRET

These values must never be exposed in client-side code.

27. Client-Side Security

The Cloudinary API secret must never be sent to the browser.

Only public configuration that is explicitly safe for client-side use may be exposed.

Sensitive credentials must remain server-side.

32. File Storage and Database

MongoDB:

User
    ↓
File Metadata
    ↓
Cloudinary publicId
    ↓
Cloudinary URL

Cloudinary:

Actual File

The database should not store the full binary file content.

33. File Storage and Backup

Cloudinary should be treated as the primary file storage service.

Important files should have appropriate backup and recovery considerations.

Database backups alone are not sufficient if the actual files are stored outside MongoDB.

36. Final File Storage Principle

Career77 file storage should be:

Secure
    ↓
Validated
    ↓
Authorized
    ↓
Organized
    ↓
Optimized
    ↓
Maintainable

Cloudinary stores the actual files, while MongoDB stores the required file metadata and references.

All file operations must follow authentication, authorization, ownership, validation, and security rules.
