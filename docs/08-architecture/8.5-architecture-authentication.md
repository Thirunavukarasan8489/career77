1. Purpose

This document defines the authentication architecture for Career77.

Authentication is responsible for:

User registration
User login
Logout
Session management
Email verification
Password recovery
Account status management
Role identification

Career77 supports three primary roles:

Candidate
Recruiter
Super Admin
2. Authentication Technology

Career77 uses:

Next.js 15+
    ↓
NextAuth.js
    ↓
Session Management
    ↓
MongoDB

Authentication should be handled securely on the server.

3. Authentication Flow

General login flow:

User
    ↓
Login
    ↓
Validate Credentials
    ↓
Authenticate User
    ↓
Create Session
    ↓
Identify Role
    ↓
Check Account Status
    ↓
Redirect to Role Portal
4. Candidate Authentication

Candidate registration flow:

Candidate
    ↓
Register
    ↓
Enter Details
    ↓
Validate Details
    ↓
Create Account
    ↓
Verify Email
    ↓
Login
    ↓
Candidate Dashboard

Candidate access should be available after successful account creation and required verification.

5. Recruiter Authentication

Recruiter registration flow:

Recruiter
    ↓
Register
    ↓
Enter Personal Details
    ↓
Enter Company Details
    ↓
Validate Details
    ↓
Create Account
    ↓
Create Company Setup
    ↓
Verification Pending
    ↓
Super Admin Review
6. Recruiter Approval Flow

If approved:

Super Admin
    ↓
Approve Recruiter
    ↓
Update Verification Status
    ↓
Send Welcome Email
    ↓
Recruiter Login
    ↓
Recruiter Dashboard
7. Recruiter Rejection Flow

If rejected:

Super Admin
    ↓
Reject Recruiter
    ↓
Recruiter Notified
    ↓
Recruiter Requests Review
    ↓
Super Admin Reviews Request
    ↓
Approve
    ↓
Send Welcome Email
    ↓
Recruiter Login
    ↓
Recruiter Dashboard

The backend should maintain the recruiter verification status.

8. Super Admin Authentication

Super Admin access should be restricted.

Flow:

Super Admin
    ↓
Login
    ↓
Authenticate
    ↓
Verify Super Admin Role
    ↓
Check Account Status
    ↓
Super Admin Dashboard

Super Admin accounts should not use the normal public recruiter registration process.

9. Login

The login process should validate:

Email
Password
Account Status
Email Verification
Role

If authentication succeeds:

Candidate
    → Candidate Portal

Recruiter
    → Recruiter Portal

Super Admin
    → Admin Portal
10. Logout

Logout should:

Invalidate Session
    ↓
Clear Authentication State
    ↓
Redirect to Public Page

The user should not be able to access protected pages after logout.

11. Session Management

Authenticated users should have a secure session.

The session should identify the authenticated user.

Relevant session information may include:

User ID
Role
Account Status

Do not store unnecessary sensitive information in the session.

12. Session Validation

Protected pages and APIs should validate the user's authentication session.

Request
    ↓
Check Session
    ↓
Session Valid?
    ├── Yes → Continue
    └── No → Login
13. Authentication and Authorization

Authentication determines:

Who is the user?

Authorization determines:

What can the user access?

Both systems should work together.

Authentication
    ↓
Identify User
    ↓
Authorization
    ↓
Check Role and Permission
14. Account Status

Accounts should have a status.

Possible statuses:

pending
active
suspended
blocked
rejected

Access should depend on the account status.

Example:

Active
    → Login Allowed

Suspended
    → Access Blocked

Blocked
    → Access Blocked

Pending
    → Limited / Verification Flow
15. Email Verification

Email verification may be required for account activation.

Flow:

Register
    ↓
Create Account
    ↓
Send Verification Email
    ↓
User Opens Link
    ↓
Verify Token
    ↓
Mark Email Verified

Verification tokens should be:

Secure
Time-limited
Single-use where appropriate
16. Password Security

Passwords must never be stored as plain text.

Passwords should be securely hashed before storage.

The database should store only the password hash where password-based authentication is used.

17. Password Reset

Password recovery flow:

Forgot Password
    ↓
Enter Email
    ↓
Generate Secure Reset Token
    ↓
Send Reset Email
    ↓
Open Reset Link
    ↓
Set New Password
    ↓
Invalidate Reset Token
    ↓
Login

Reset tokens should expire after a limited period.

18. Password Reset Security

The system should avoid revealing whether an email address exists.

Example response:

If an account exists, a password reset email has been sent.

This helps prevent account enumeration.

19. Protected Routes

Protected areas include:

/candidate/*
/recruiter/*
/admin/*

Unauthenticated users should be redirected to login.

20. Role-Based Route Access

Candidate:

/candidate/*

Recruiter:

/recruiter/*

Super Admin:

/admin/*

Users should not access another role's protected portal.

21. API Authentication

Protected APIs must verify authentication.

Example:

GET /api/candidates/me

The server should identify the current authenticated user from the session.

Do not trust a user ID supplied by the client for ownership checks.

22. Resource Ownership

Authentication should work together with ownership validation.

Example:

Candidate
    ↓
Request Own Application
    ✓ Allowed

Candidate
    ↓
Request Another Candidate's Application
    ✗ Forbidden

Authentication alone is not enough.

23. Recruiter Account Verification

Recruiter access should depend on verification status.

Example:

Recruiter Registered
    ↓
Pending Verification
    ↓
Super Admin Review

Before approval:

Recruiter Portal Access
    ↓
Restricted

After approval:

Verified
    ↓
Recruiter Portal Access
24. Recruiter Review Request

If a recruiter is rejected:

Rejected
    ↓
Request Review
    ↓
Super Admin Review
    ↓
Decision

The system should maintain the review request state.

25. Welcome Email

After recruiter approval:

Recruiter Approved
    ↓
Verification Status Updated
    ↓
Welcome Email Sent
    ↓
Recruiter Login

The email should inform the recruiter that their account is approved.

26. Authentication Middleware

Next.js middleware may be used for initial route protection.

Example:

Request
    ↓
Middleware
    ↓
Check Session
    ↓
Protected Route?
    ├── No → Continue
    └── Yes
         ↓
      Authenticated?
         ├── Yes → Continue
         └── No → Redirect Login

Final authorization must still be enforced on the server.

27. Server-Side Protection

Sensitive operations must be protected server-side.

Examples:

Approve Recruiter
Reject Recruiter
Create Job
Update Job
View Applicants
Change Application Status
Process Payment
Manage Users

Never rely only on frontend route protection.

28. Authentication Cookies

Authentication cookies should use secure settings.

Where applicable:

HttpOnly
Secure
SameSite

Cookie configuration should follow the authentication library's recommended secure configuration.

29. Session Expiration

Sessions should have appropriate expiration policies.

The exact duration should be determined based on:

Security
User Experience
Role
Platform Requirements

Sensitive administrative sessions may require stricter policies.

30. Account Blocking

If an account is blocked or suspended:

Account Status Changed
    ↓
Existing Session Checked
    ↓
Access Revoked

Blocked users should not continue accessing protected resources.

31. Login Security

The login system should consider:

Rate limiting
Failed login monitoring
Secure password handling
Session protection
Suspicious activity detection

Sensitive authentication events should be logged securely.

32. Authentication Errors

Authentication errors should be user-friendly.

Avoid exposing internal information.

Instead of:

User exists in MongoDB but password hash mismatch.

Use:

Invalid email or password.

35. Security Principles

Career77 authentication should follow:

Secure Passwords
    ↓
Secure Sessions
    ↓
Verified Accounts
    ↓
Role Validation
    ↓
Server-Side Authorization
    ↓
Resource Ownership
    ↓
Audit Logging
36. Final Authentication Principle

Career77 authentication should provide a secure and simple login experience while maintaining strict separation between Candidate, Recruiter, and Super Admin access.

The system must ensure:

User Identity
    +
Account Status
    +
Role
    +
Permission
    +
Resource Ownership

are validated before allowing access to protected Career77 resources.
