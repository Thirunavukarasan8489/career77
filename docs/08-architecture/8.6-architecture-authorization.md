1. Purpose

This document defines the authorization architecture for Career77.

Authorization controls what an authenticated user is allowed to access and perform.

Career77 has three primary roles:

Candidate
Recruiter
Super Admin
2. Authentication vs Authorization

Authentication determines:

Who is the user?

Authorization determines:

What is the user allowed to do?

Career77 must perform both checks.

User Request
    ↓
Authentication
    ↓
Identify User
    ↓
Authorization
    ↓
Check Role
    ↓
Check Permission
    ↓
Check Resource Ownership
    ↓
Allow or Deny
3. Authorization Principles

Career77 authorization should follow:

Least privilege
Server-side enforcement
Role-based access
Resource ownership
Permission checks
Account status checks

The frontend should never be the only layer responsible for authorization.

4. Primary Roles

Career77 supports:

Candidate
Recruiter
Super Admin

Each role has a separate responsibility and access scope.

5. Candidate Authorization

Candidates can access their own:

Profile
Resume
Applications
Interviews
Training Enrollments
Payments
Notifications
Settings

Example:

Candidate
    ↓
Own Profile
    ✓ Allowed

Candidate
    ↓
Another Candidate's Profile
    ✗ Denied
6. Recruiter Authorization

Recruiters can access resources belonging to companies they are authorized to manage.

Examples:

Company
Jobs
Applicants
Hiring Pipeline
Interviews
Recruiter Analytics

Example:

Recruiter A
    ↓
Company A Jobs
    ✓ Allowed

Recruiter A
    ↓
Company B Jobs
    ✗ Denied
7. Super Admin Authorization

Super Admin has platform-level administrative access.

Super Admin may manage:

Users
Candidates
Recruiters
Companies
Jobs
Verification
Analytics
CMS
Billing
Subscriptions
Support
System Settings
Audit Logs

Super Admin access should be strictly protected.

8. Account Status Authorization

Authorization should consider account status.

Example:

Active
    ↓
Access Allowed

Pending
    ↓
Limited Access

Suspended
    ↓
Access Denied

Blocked
    ↓
Access Denied

Rejected
    ↓
Access Denied
9. Recruiter Verification Authorization

Recruiter access depends on verification.

Flow:

Recruiter Registration
    ↓
Pending
    ↓
Super Admin Review

If approved:

Verified
    ↓
Recruiter Portal Access

If rejected:

Rejected
    ↓
Recruiter Portal Access Restricted

If review is requested:

Review Requested
    ↓
Super Admin Review
10. Route Authorization

Protected route groups:

/candidate/*
/recruiter/*
/admin/*

Access rules:

Candidate
    → /candidate/*

Recruiter
    → /recruiter/*

Super Admin
    → /admin/*

Users should not access another role's protected routes.

11. API Authorization

Every protected API should validate authorization.

Example:

PATCH /api/jobs/[id]

The server should verify:

User Authenticated?
    ↓
User Is Recruiter?
    ↓
Recruiter Is Verified?
    ↓
Recruiter Owns / Manages Company?
    ↓
Job Belongs to Company?
    ↓
Allow Update
12. Resource Ownership

Resource ownership is a key authorization rule.

Example:

Candidate
    ↓
Application
    ↓
Job
    ↓
Company

The candidate can access their own application.

A recruiter can access applications for jobs belonging to their authorized company.

A Super Admin can access platform-wide application data.

16. Permission Checks

Authorization checks may include:

Is Authenticated?
Is Correct Role?
Is Account Active?
Is Recruiter Verified?
Does User Own Resource?
Does User Manage Company?
Does User Have Permission?

The backend should perform the relevant checks before executing the operation.

17. Authorization Middleware

Middleware may provide initial route protection.

Example:

Request
    ↓
Middleware
    ↓
Check Session
    ↓
Check Route
    ↓
Allow / Redirect

However, middleware should not replace server-side authorization.

18. Server-Side Authorization

Sensitive operations must always be authorized on the server.

Examples:

Approve Recruiter
Reject Recruiter
Publish Job
View Applicants
Update Application Status
Manage Users
Process Refund
Change Subscription
19. Recruiter Job Authorization

Recruiter job operations should follow:

Recruiter
    ↓
Authenticated?
    ↓
Verified?
    ↓
Company Access?
    ↓
Job Ownership?
    ↓
Perform Action

Actions may include:

Create
View
Update
Pause
Publish
Close
Delete
20. Applicant Authorization

Recruiters should only access applicants for jobs they are authorized to manage.

Recruiter
    ↓
Company
    ↓
Job
    ↓
Applications
    ↓
Candidates

The backend must verify the entire relationship.

21. Application Authorization

Candidates:

View Own Application
Update Allowed Candidate Fields
Withdraw Application

Recruiters:

View Authorized Applications
Update Application Status
Add Recruiter Notes
Schedule Interviews

Super Admin:

Platform-Level Access
22. Interview Authorization

Interview access should be restricted to relevant users.

Candidate
    ↓
Own Interview
Recruiter
    ↓
Interview For Authorized Job
Super Admin
    ↓
Administrative Access
23. Training Authorization

Training access should be controlled based on enrollment.

Public User
    ↓
View Published Training
Candidate
    ↓
Purchase Training
    ↓
Payment Verified
    ↓
Enrollment Created
    ↓
Training Access

Candidates should not access paid training content without valid enrollment.

24. Payment Authorization

Payment operations should be protected.

Candidate
    ↓
Create Own Payment

Users should not be able to modify another user's payment records.

Payment verification must be performed by the server.

25. Subscription Authorization

Recruiter subscription access should be based on:

Authenticated User
    ↓
Recruiter Role
    ↓
Company Relationship
    ↓
Subscription Ownership

Users should not modify another company's subscription.

26. Super Admin Verification Authorization

Only authorized Super Admin users should be able to:

Approve Recruiter
Reject Recruiter
Review Rejected Recruiter
Suspend Recruiter
Reactivate Recruiter

Every important action should be recorded in audit logs.

27. CMS Authorization

CMS operations should be restricted to Super Admin.

Examples:

Create Content
Update Content
Publish Content
Unpublish Content
Delete Content

Public users may only view published content.

28. Billing Authorization

Billing data should be protected.

Recruiter
    ↓
Own Company Billing
Super Admin
    ↓
Platform Billing

Users should not access billing data outside their authorized scope.

29. Support Authorization

Support access should follow:

Candidate
    → Own Tickets

Recruiter
    → Own / Company Tickets

Super Admin
    → Platform Support
30. Authorization Failure

If a user is not authorized:

401 Unauthorized

should be used when authentication is missing.

403 Forbidden

should be used when the user is authenticated but does not have permission.

31. Frontend Authorization

The frontend may hide or show features based on role.

Example:

Candidate
    → Candidate Navigation

Recruiter
    → Recruiter Navigation

Super Admin
    → Admin Navigation

However:

Frontend Authorization
    ≠
Security Authorization

The backend must always enforce the final authorization decision.

Career77 authorization must follow:

Never Trust Client
    ↓
Always Verify Server-Side
    ↓
Least Privilege
    ↓
Check Ownership
    ↓
Check Role
    ↓
Check Account Status
    ↓
Audit Sensitive Actions
38. Final Authorization Principle

Career77 authorization should ensure that every user can access only the resources and actions appropriate to their role and ownership.

The authorization model is:

Candidate
    → Own Data and Own Career Activities

Recruiter
    → Authorized Company and Recruitment Activities

Super Admin
    → Platform Administration

All sensitive operations must be protected by server-side authorization, with resource ownership and account status checked before access is granted.
