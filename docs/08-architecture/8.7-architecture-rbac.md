1. Purpose

This document defines the Role-Based Access Control (RBAC) architecture for Career77.

RBAC controls access to platform features based on the user's assigned role.

Career77 has three primary roles:

Candidate
Recruiter
Super Admin
2. RBAC Model

Career77 follows:

User
    ↓
Role
    ↓
Permissions
    ↓
Resource Access

Example:

Recruiter
    ↓
Recruiter Permissions
    ↓
Company + Jobs + Applicants

9. Role Assignment

A user should have a primary role.

Example:

User
    ↓
role: candidate
User
    ↓
role: recruiter
User
    ↓
role: super_admin

The role should be stored and validated server-side.

10. Candidate RBAC Flow
Candidate Login
    ↓
Role = Candidate
    ↓
Load Candidate Permissions
    ↓
Access Candidate Portal

Recruiter Portal
Admin Portal

Platform Administration
11. Recruiter RBAC Flow
Recruiter Login
    ↓
Role = Recruiter
    ↓
Check Account Status
    ↓
Check Verification Status
    ↓
Load Recruiter Permissions
    ↓
Access Recruiter Portal

Unverified recruiters should have restricted access.

12. Super Admin RBAC Flow
Super Admin Login
    ↓
Role = Super Admin
    ↓
Check Account Status
    ↓
Load Admin Permissions
    ↓
Access Admin Portal

Super Admin access should be strictly protected.

13. Recruiter Verification and RBAC

Recruiter role alone should not automatically grant full recruiter access.

The system should check:

Role
    +
Account Status
    +
Verification Status

Example:

Role = Recruiter
Status = Active
Verification = Verified

Result:

Full Recruiter Portal Access

17. Resource-Level Authorization

RBAC should be combined with resource ownership.

Example:

Recruiter
    ↓
job.update
    ↓
Check Job
    ↓
Does Job Belong to Recruiter's Company?
    ↓
Yes → Allow
No  → Deny

Having the job.update permission alone is not sufficient.

18. Company-Level Access

Recruiters should only manage authorized companies.

Recruiter
    ↓
Company
    ↓
Authorized?
    ├── Yes → Access
    └── No → Deny

This prevents cross-company data access.

19. Applicant Access

Recruiter applicant access:

Recruiter
    ↓
applicant.view
    ↓
Check Job
    ↓
Check Company
    ↓
Check Recruiter Authorization
    ↓
Allow Access

Recruiters should not view applicants belonging to unrelated companies.

20. Candidate Data Access

Candidates can access their own private data.

Candidate
    ↓
candidateId = currentUser.id
    ↓
Allow

Requests for another candidate's private information should be denied.

21. Super Admin Override

Example:

Super Admin actions should still be audited.

23. Frontend Permission Checks

Frontend checks improve user experience.

Example:

if (user.role === "recruiter") {
  showRecruiterNavigation();
}

However, frontend checks must never be treated as security controls.

26. Permission Denied

If the user lacks permission:

403 Forbidden

Example response:

{
  "success": false,
  "message": "You do not have permission to perform this action."
}
27. RBAC Service

Authorization logic should be centralized where practical.

Example structure:

lib/
└── auth/
    ├── permissions.ts
    ├── roles.ts
    └── authorization.ts

This avoids duplicating authorization logic throughout the application.

28. Permission Constants

Permissions should be defined consistently.

Example:

PERMISSIONS = {
  JOB_CREATE: "job.create",
  JOB_VIEW: "job.view",
  JOB_UPDATE: "job.update",
  JOB_DELETE: "job.delete",
  APPLICATION_VIEW: "application.view",
}

This reduces spelling mistakes and inconsistent permission names.

29. Role Constants

Roles should also use centralized constants.

ROLES = {
  CANDIDATE: "candidate",
  RECRUITER: "recruiter",
  SUPER_ADMIN: "super_admin",
}

Avoid hardcoding role strings throughout the application.

30. RBAC and Account Status

All applicable checks should pass before allowing access.

32. RBAC Security Rules

Career77 must follow these rules:

Never trust role information from the client.
Always validate roles on the server.
Always verify resource ownership.
Check recruiter verification status.
Check account status.
Protect Super Admin operations.
Record sensitive administrative actions.
Apply least-privilege access.
Deny access by default.
Keep authorization logic centralized.

34. Final RBAC Principle

Career77 RBAC should provide clear and secure separation between the three primary platform roles.

Candidate
    ↓
Manage Own Career Activities

Recruiter
    ↓
Manage Authorized Recruitment Activities

Super Admin
    ↓
Manage Career77 Platform

RBAC must always be enforced on the server side, combined with resource ownership, account status, recruiter verification, and permission checks to ensure that users can access only the data and functionality they are authorized to use.
