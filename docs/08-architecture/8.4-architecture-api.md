1. Purpose

This document defines the API architecture for Career77.

The API layer connects the frontend with backend business logic and data services.

Frontend
    ↓
API
    ↓
Authentication
    ↓
Authorization
    ↓
Validation
    ↓
Business Logic
    ↓
Database / External Services
2. API Technology

Career77 APIs are built using:

Next.js 15+
    ↓
App Router
    ↓
Route Handlers
    ↓
TypeScript
    ↓
Mongoose
    ↓
MongoDB
3. API Base Path

All application APIs should use:

/api

Example:

/api/jobs
/api/applications
/api/candidates
/api/recruiters

APIs should be organized by business domain.

5. Authentication APIs

Authentication endpoints may include:

POST /api/auth/register
POST /api/auth/login
POST /api/auth/logout
POST /api/auth/forgot-password
POST /api/auth/reset-password
POST /api/auth/verify-email

NextAuth.js manages the authentication session where applicable.

6. User APIs

Example endpoints:

GET    /api/users/me
PATCH  /api/users/me
DELETE /api/users/me

Users should only be able to access or modify their own account information unless authorized otherwise.

7. Candidate APIs

Example:

GET    /api/candidates/me
PATCH  /api/candidates/me
GET    /api/candidates/me/applications
GET    /api/candidates/me/interviews
GET    /api/candidates/me/trainings

Candidate APIs must enforce candidate ownership.

8. Recruiter APIs

Example:

GET    /api/recruiters/me
PATCH  /api/recruiters/me
GET    /api/recruiters/me/company

Recruiter access should depend on account verification and authorization.

9. Company APIs

Example:

GET    /api/companies
GET    /api/companies/[id]
POST   /api/companies
PATCH  /api/companies/[id]

Company modification APIs should require appropriate recruiter or Super Admin permissions.

10. Job APIs

Public job search:

GET /api/jobs

Job details:

GET /api/jobs/[id]

Recruiter job management:

POST   /api/jobs
PATCH  /api/jobs/[id]
DELETE /api/jobs/[id]
11. Job Search Parameters

Job search may support:

keyword
experience
location
jobType
salary
company
page
limit
sort

Example:

/api/jobs?keyword=react&experience=mid&location=coimbatore

Search and filtering should remain consistent with the public job search experience.

12. Job Publishing

Job publishing should follow authorization rules.

Example flow:

Recruiter
    ↓
Create Job
    ↓
Save Draft
    ↓
Submit for Review
    ↓
Super Admin Review
    ↓
Approve
    ↓
Publish

If the product later allows automatic publishing for verified recruiters, this flow can be updated.

13. Application APIs

Example:

POST /api/applications
GET  /api/applications/[id]
PATCH /api/applications/[id]

Candidate application flow:

Candidate
    ↓
Select Job
    ↓
Apply
    ↓
Validate
    ↓
Check Job Status
    ↓
Check Duplicate Application
    ↓
Create Application
14. Application Status APIs

Recruiters may update application status.

Example:

PATCH /api/applications/[id]/status

Possible statuses:

applied
screening
shortlisted
interview
offer
hired
rejected
withdrawn

The backend must verify that the recruiter has access to the application.

15. Interview APIs

Example:

POST   /api/interviews
GET    /api/interviews/[id]
PATCH  /api/interviews/[id]
DELETE /api/interviews/[id]

Interview data should be accessible only to authorized users.

16. Training APIs

Example:

GET  /api/trainings
GET  /api/trainings/[id]
POST /api/trainings
PATCH /api/trainings/[id]

Public users may view published training programs.

Training management should require appropriate administrative permissions.

17. Enrollment APIs

Example:

POST /api/enrollments
GET  /api/enrollments/[id]
GET  /api/enrollments/me

Enrollment should normally be created after successful payment verification for paid training.

18. Payment APIs

Example:

POST /api/payments/create
POST /api/payments/verify
GET  /api/payments/[id]

Payment verification must always happen server-side.

The frontend must never be trusted to confirm payment success.

19. Subscription APIs

If recruiter subscriptions are enabled:

GET  /api/subscriptions
POST /api/subscriptions
PATCH /api/subscriptions/[id]
POST /api/subscriptions/[id]/cancel

Subscription access should be determined by server-side subscription status.

20. Notification APIs

Example:

GET   /api/notifications
PATCH /api/notifications/[id]/read
PATCH /api/notifications/read-all

Notifications may be generated by backend business events.

21. Support APIs

Example:

POST  /api/support/tickets
GET   /api/support/tickets
GET   /api/support/tickets/[id]
PATCH /api/support/tickets/[id]

Users should only access their own support tickets unless they are Super Admins or assigned support staff.

22. Super Admin APIs

Super Admin APIs may include:

GET   /api/admin/users
PATCH /api/admin/users/[id]

GET   /api/admin/companies
PATCH /api/admin/companies/[id]

GET   /api/admin/jobs
PATCH /api/admin/jobs/[id]

GET   /api/admin/verification
PATCH /api/admin/verification/[id]

GET   /api/admin/analytics

All Admin APIs must require Super Admin authorization.

23. Recruiter Verification API

Recruiter verification flow:

GET /api/admin/recruiters/pending

Approve:

PATCH /api/admin/recruiters/[id]/approve

Reject:

PATCH /api/admin/recruiters/[id]/reject

Review request:

PATCH /api/admin/recruiters/[id]/review-request

The backend should record important verification actions in audit logs.

24. API Request Flow

Every protected API should follow:

Incoming Request
    ↓
Authenticate
    ↓
Identify User
    ↓
Check Role
    ↓
Check Permission
    ↓
Validate Request
    ↓
Execute Service
    ↓
Database Operation
    ↓
Return Response
25. Request Validation

Validate all incoming data.

Validation applies to:

Body
Query Parameters
Route Parameters
File Uploads

Invalid requests should be rejected before business logic is executed.

26. Response Format

API responses should follow a consistent structure.

Success:

{
  "success": true,
  "data": {}
}

Success with message:

{
  "success": true,
  "message": "Job created successfully",
  "data": {}
}

Error:

{
  "success": false,
  "message": "Unable to create job"
}

The exact response structure should remain consistent across APIs.

27. HTTP Status Codes

Use appropriate status codes.

200 OK
201 Created
204 No Content
400 Bad Request
401 Unauthorized
403 Forbidden
404 Not Found
409 Conflict
422 Unprocessable Entity
429 Too Many Requests
500 Internal Server Error
28. Error Handling

Do not expose internal errors to clients.

Avoid returning:

Database Connection Failed at line 45

Return a safe message:

Unable to process your request.

Detailed errors should be logged securely on the server.

29. Authentication and Authorization

API protection should distinguish:

Authentication
    ↓
Is the user logged in?

Authorization
    ↓
Is the user allowed to perform this action?

Both must be enforced where required.

30. Resource Ownership

APIs must verify resource ownership.

Example:

Candidate A
    ↓
Request Application A
    ✓ Allowed

Candidate A
    ↓
Request Application B
    ✗ Forbidden

The same principle applies to recruiter company and job access.

31. Pagination

List APIs should support pagination.

Example:

/api/jobs?page=1&limit=20

Response may include:

{
  "success": true,
  "data": [],
  "pagination": {
    "page": 1,
    "limit": 20,
    "total": 100,
    "totalPages": 5
  }
}
32. Filtering and Sorting

List APIs should support controlled filters.

Example:

/api/jobs?status=published&experience=mid

Sorting:

/api/jobs?sort=createdAt&order=desc

Only supported fields should be accepted.

37. API and Search

Job search API should initially use MongoDB text indexes.

Example:

/api/jobs?keyword=full+stack+developer

The search API should support filters and pagination.

If search requirements grow, the backend may migrate to MongoDB Atlas Search.

38. API Versioning

Versioning is not required for the initial release if the API is controlled within the Career77 application.

If external consumers or major breaking changes are introduced later, versioning may be considered.

Possible future structure:

/api/v1/jobs
/api/v2/jobs

43. API Architecture Principle

Career77 APIs should be:

Consistent
    ↓
Secure
    ↓
Validated
    ↓
Role-Aware
    ↓
Resource-Aware
    ↓
Scalable
    ↓
Maintainable

All APIs should provide a consistent foundation for the Candidate, Recruiter, and Super Admin experiences while securely connecting Career77's frontend, database, payments, file storage, notifications, and other platform services.
