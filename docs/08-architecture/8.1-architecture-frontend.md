1. Purpose

This document defines the frontend architecture for Career77.

The frontend is responsible for:

Public job discovery
Candidate experience
Recruiter experience
Super Admin experience
Authentication UI
Training experience
Payment UI
Responsive user interfaces
2. Frontend Technology

Career77 frontend uses:

Next.js 15+
    ↓
App Router
    ↓
React
    ↓
TypeScript
    ↓
Tailwind CSS

Deployment:

Vercel
3. Rendering Strategy

Career77 should use the appropriate Next.js rendering strategy.

Static Rendering
    ↓
Static Public Content

Dynamic Rendering
    ↓
Authenticated / Personalized Content

ISR
    ↓
Content That Changes Periodically

Client Components
    ↓
Interactive UI

Use Server Components by default.

Use Client Components only when client-side interaction is required.

4. App Router

The application should use the Next.js App Router.

Recommended route organization:

app/
├── (public)/
├── (auth)/
├── (candidate)/
├── (recruiter)/
├── (admin)/
└── api/

Route groups should organize application areas without unnecessarily affecting URLs.

11. Component Organization

Recommended structure:

components/
├── ui/
├── forms/
├── navigation/
├── cards/
├── tables/
├── dialogs/
├── feedback/
├── public/
├── candidate/
├── recruiter/
└── admin/

Reusable components should remain independent of specific pages whenever possible.

12. Server Components

Use Server Components by default.

Suitable for:

Public pages
Job details
Company profiles
Data fetching
SEO content
Static content

Benefits:

Reduced client JavaScript
Better performance
Server-side data access
Better initial rendering
13. Client Components

Use Client Components when required for:

User interaction
Form state
Browser APIs
Event handlers
Interactive filters
Modals
Drag and drop
Real-time UI

Avoid marking entire pages as Client Components unnecessarily.

14. Data Fetching

Data fetching should be placed as close as practical to the component or route that needs the data.

Use:

Server Components
    ↓
Server-side Data Fetching

For highly interactive client-side workflows, use appropriate client-side data fetching patterns.

15. API Integration

Frontend communication with backend APIs should use a consistent API layer.

Recommended:

UI
    ↓
Feature Hook / Service
    ↓
API Client
    ↓
Backend API

Avoid placing repeated API request logic directly inside multiple components.

16. State Management

Use the simplest state management solution appropriate for the requirement.

Recommended:

Local State
    ↓
Component-specific State

URL State
    ↓
Search and Filter State

Server State
    ↓
Fetched Data

Global State
    ↓
Shared Application State

Avoid unnecessary global state.

17. URL State

Use URL parameters for shareable search and filter states where appropriate.

Example:

/jobs?keyword=developer&experience=mid&location=coimbatore

This allows users to:

Share search results
Bookmark searches
Use browser navigation
Preserve search state
18. Forms

Forms should use reusable components and consistent validation.

Common forms:

Candidate Registration
Candidate Profile
Recruiter Registration
Company Setup
Create Job
Apply Job
Training Enrollment
Payment
Settings
19. Form Validation

Frontend validation should provide immediate feedback.

Server-side validation must also be performed.

Frontend Validation
    ↓
User Feedback

Backend Validation
    ↓
Security and Data Integrity

Never rely only on frontend validation.

20. Authentication Integration

The frontend should integrate with NextAuth.js.

Authentication state should control access to:

Candidate Portal
Recruiter Portal
Super Admin Portal

The frontend should not rely only on UI hiding for authorization.

Backend authorization is required.

21. Role-Based Routing

The application should support:

Candidate
Recruiter
Super Admin

After authentication:

User
    ↓
Identify Role
    ↓
Load Role-Specific Experience

Unauthorized users should not access protected routes.

22. Middleware

Next.js middleware may be used for:

Authentication checks
Route protection
Redirects
Request-level logic

Sensitive authorization decisions must still be enforced on the server.

23. Layout Architecture

Use shared layouts for each major application area.

PublicLayout
AuthLayout
CandidateLayout
RecruiterLayout
AdminLayout

Each layout should provide consistent navigation and structure.

24. Error Handling

Use consistent error handling.

Provide:

Loading State
Empty State
Error State
Success State

Users should receive clear recovery actions.

25. Loading UI

Use:

Skeletons
Loading indicators
Button loading states

Avoid unnecessary full-page loading indicators.

26. Error Boundaries

Major application sections should have appropriate error boundaries.

Examples:

Public Pages
Candidate Portal
Recruiter Portal
Admin Portal

Errors should not unnecessarily break the entire application.

27. SEO

Public pages should be optimized for search engines.

Focus on:

Page metadata

Important public pages include:

Job Details

37. Feature Organization

Complex features may use feature-based organization.

Example:

features/
├── jobs/
├── applications/
├── candidates/
├── recruiters/
├── training/
├── payments/
└── notifications/

The project should choose a consistent organization approach and avoid unnecessary duplication.

38. API Routes

Next.js API routes should be used only where appropriate.

Backend business logic should remain organized and reusable.

Avoid placing large business logic directly inside route handlers.

39. Security

Frontend security practices include:

Never expose secrets
Never trust client-side authorization
Validate user input
Avoid unsafe HTML rendering
Protect authenticated routes
Use secure authentication flows

Secrets must remain server-side.

40. Environment Variables

Use environment variables for configuration.

Examples:

DATABASE_URI
NEXTAUTH_SECRET
NEXTAUTH_URL
CLOUDINARY_CLOUD_NAME
CLOUDINARY_API_KEY
CLOUDINARY_API_SECRET

Secret values must never be committed to source control.

41. Logging

Avoid logging sensitive information.

Never log:

Passwords
Authentication secrets
Payment credentials
Private tokens

Use structured logging for important server-side events.

42. Testing

Frontend testing should cover:

Unit Tests
Component Tests
Integration Tests
End-to-End Tests
Accessibility Tests

Critical workflows should receive priority.
