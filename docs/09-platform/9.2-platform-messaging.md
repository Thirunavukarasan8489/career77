1. Purpose

This document defines the messaging architecture for Career77.

Messaging allows relevant users to communicate during recruitment and support workflows.

Career77 messaging should be simple, secure, and focused on professional communication.

2. Messaging Users

Messaging may be available between:

Candidate ↔ Recruiter
Candidate ↔ Support
Recruiter ↔ Support
Super Admin ↔ Candidate
Super Admin ↔ Recruiter

Users should only communicate within authorized contexts.

4. Conversation Types

Career77 may support:

Recruitment Conversation
Support Conversation
Administrative Conversation
5. Recruitment Messaging

Recruitment messaging connects candidates and recruiters.

Example:

Candidate
    ↓
Applied for Job
    ↓
Recruiter
    ↓
Conversation

Messaging may be enabled when the candidate has a valid application or when the recruiter initiates contact through an authorized workflow.

6. Candidate Messaging

Candidates may:

View authorized conversations
Send messages
Receive messages
Reply to recruiters
Contact support

Candidates should not be able to initiate unrestricted conversations with every recruiter on the platform.

7. Recruiter Messaging

Recruiters may:

View conversations related to their authorized jobs
Contact applicants
Reply to candidates
Send interview-related messages
Contact support

Recruiters should only access conversations related to their authorized company and recruitment activities.

8. Super Admin Messaging

Super Admin may access administrative conversations when required.

Examples:

Recruiter Verification
Account Review
Support Escalation
Platform Issues
Compliance Matters

Super Admin access should be controlled and audited.

18. Attachment Authorization

Attachments must follow the same access rules as the conversation.

Candidate

19. Real-Time Messaging

Real-time messaging may be introduced when required.

Possible architecture:

User
    ↓
WebSocket / Real-Time Service
    ↓
Messaging Service
    ↓
Database

The initial implementation may use standard API-based messaging if real-time communication is not required at launch.

20. Message Pagination

Messages should use pagination.

Example:

GET /api/conversations/[id]/messages?page=1&limit=30

Older messages can be loaded as the user scrolls.

21. Conversation Pagination

Conversation lists should also use pagination.

Example:

GET /api/conversations?page=1&limit=20

This prevents large conversation histories from being loaded at once.

Example:

Possible controls:

31. Message Deletion

The system may support:

36. Security Rules

Career77 messaging must follow:

Users must be authenticated.
Users must be authorized to access conversations.
Users must only access permitted messages.
Recruiters must only access authorized company conversations.
Candidates must only access their own conversations.
Attachments must follow file authorization rules.
Messages must be validated and sanitized.
Rate limiting should prevent spam.
Sensitive actions should be audited.
Suspended or blocked users should have restricted messaging access.
37. Final Messaging Principle

Career77 messaging should support professional communication between candidates, recruiters, and support teams without exposing unauthorized data.

The system should be:

Secure
    ↓
Private
    ↓
Context-Aware
    ↓
Authorized
    ↓
Scalable

Messaging access should always be based on authentication, role, conversation membership, resource ownership, and business context.
