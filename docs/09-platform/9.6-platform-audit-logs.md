1. Purpose

This document defines the audit log architecture for Career77.

Audit logs record important actions performed across the platform.

Audit logs help with:

Security
Accountability
Troubleshooting
Compliance
Administrative monitoring
Business activity tracking
2. Audit Log Architecture
User Action
    ↓
Application Service
    ↓
Business Operation
    ↓
Audit Log Service
    ↓
MongoDB

The audit log should be generated by the backend after important operations.

3. Audit Log Users

Audit logs primarily support:

Candidate
Recruiter
Super Admin

Super Admin has access to the platform audit log dashboard according to permissions.

Candidates and recruiters should generally not have direct access to the global audit log system.

  "createdAt": "..."

13. Actor

The actor identifies who performed the action.

Possible actors:

Candidate
Recruiter
Super Admin
System

A system-generated action should use:

actorType = system

when no direct user initiated the action.

14. Action

Actions should use consistent names.

Example:

USER_LOGIN
USER_LOGOUT
PROFILE_UPDATED
JOB_CREATED
JOB_PUBLISHED
APPLICATION_SUBMITTED
PAYMENT_SUCCESS
SUBSCRIPTION_CANCELLED
TICKET_RESOLVED

Using standardized action names makes filtering and reporting easier.

15. Resource Type

The resource type identifies what was affected.

Examples:

user
company
job
application
training
payment
subscription
support_ticket
file
16. Resource ID

The resource ID identifies the specific affected record.

Example:

resourceType = job
resourceId = 123456

This allows administrators to trace an action back to the relevant resource.

17. Audit Metadata

Metadata may contain additional information.

Example:

{
  "previousStatus": "pending",
  "newStatus": "approved"
}

Metadata should only contain information useful for auditing.

Avoid storing unnecessary sensitive data.

18. IP Address

The system may record the IP address for security-sensitive events.

Examples:

Login
Password Change
Account Security Changes
Administrative Actions

IP storage must follow applicable privacy and data protection requirements.

19. User Agent

The system may record the browser or client information for security investigations.

Example:

Chrome
Firefox
Safari
Mobile App

This information should be used only when it provides legitimate security or operational value.

20. Audit Log Immutability

Audit logs should generally be append-only.

Create Audit Log
    ↓
Store
    ↓
Do Not Modify

Normal users should not be able to edit or delete audit records.

21. Audit Log Deletion

Audit logs should not be casually deleted.

If retention requirements require deletion, it should be performed through a controlled administrative process.

Any audit-log deletion or archival operation should itself be audited.

22. Audit Log Access

Super Admin may access audit logs according to authorization.

Example:

28. Audit Log and Authentication

Authentication events should be recorded carefully.

Example:

User
    ↓
Login Attempt
    ↓
Authentication Service
    ↓
Success / Failure
    ↓
Audit Log

Failed login attempts may be useful for detecting suspicious activity.

29. Audit Log and Payments

Payment information should be logged at an event level.

Example:

Payment
    ↓
Verified
    ↓
Audit Log

The audit record may include:

Payment ID
Transaction ID
Status
Actor / System
Timestamp

It must not contain:

Card Number
CVV
Payment Credentials
30. Audit Log and File Storage

Important file actions may be audited.

Examples:

Resume Uploaded
Resume Deleted
Certificate Uploaded
Company Logo Updated

The audit record should reference the file or resource ID rather than storing the actual file.

31. Audit Log Service

Audit logic should be centralized.

Recommended structure:

lib/
└── audit/
    ├── audit-service.ts
    ├── audit-actions.ts
    └── audit-types.ts
32. Audit Service Example Flow
Business Operation
    ↓
Success
    ↓
Audit Service
    ↓
Create Audit Record
    ↓
MongoDB

Example:

Recruiter
    ↓
Publishes Job
    ↓
Job Updated
    ↓
Audit Log Created
33. Audit Logging Failure

Audit logging should not normally cause a successful business operation to fail.

Example:

Job Published
    ↓
Job Saved Successfully
    ↓
Audit Log Failed

The job should remain published.

However, the audit failure should be logged through the system's operational monitoring mechanisms.

For critical compliance workflows, stricter transaction or event-processing guarantees may be required.

34. Audit Log API

Possible API structure:

GET /api/admin/audit-logs
GET /api/admin/audit-logs/[id]

Audit log creation should normally be internal.

The frontend should not directly create audit records.

35. Audit Log Security

Career77 should:

Make audit logs append-only
Restrict access to authorized administrators
Protect sensitive information
Validate administrator permissions
Record important administrative actions
Prevent normal users from modifying logs
Monitor suspicious activity
36. Audit Log Retention

Career77 should define an audit log retention policy.

The retention period may depend on:

Security Requirements
Business Requirements
Legal Requirements
Privacy Requirements

The exact retention period should be defined separately.

39. Final Audit Log Principle

Career77 audit logs should provide a reliable record of important platform activity.

The audit system should be:

Secure
    ↓
Immutable
    ↓
Traceable
    ↓
Searchable
    ↓
Role-Based
    ↓
Privacy-Aware
    ↓
Auditable

The primary goal is to answer:

Who performed what action, on which resource, when, and from where?

Audit logging should be implemented consistently across authentication, recruiter verification, jobs, applications, training, payments, subscriptions, support, file management, and Super Admin operations.
