1. Purpose

This document defines the payment architecture for Career77.

Career77 may process payments for:

Paid training
Training enrollment
Recruiter subscriptions
Other future paid platform services

All payment operations must be securely processed through a supported payment gateway.

2. Payment Architecture
User
    ↓
Select Product / Service
    ↓
Create Payment Request
    ↓
Payment Gateway
    ↓
User Completes Payment
    ↓
Gateway Response / Webhook
    ↓
Verify Payment Server-Side
    ↓
Update Payment Status
    ↓
Activate Service
    ↓
Send Notification
3. Payment Users

Primary payment users:

    ↓
Paid Training

    ↓
Subscription / Paid Services

Super Admin
    ↓
Payment Management
4. Candidate Training Payment

Candidates can purchase paid training.

Flow:

Candidate
    ↓
View Training
    ↓
Select Training
    ↓
Click Enroll / Buy Now
    ↓
Create Payment
    ↓
Payment Gateway
    ↓
Payment Success
    ↓
Verify Payment
    ↓
Create Enrollment
    ↓
Grant Training Access
5. Payment Verification

The frontend payment response must not be treated as final proof of payment.

The backend must verify the transaction.

Payment Gateway
    ↓
Webhook / Verification API
    ↓
Backend
    ↓
Verify Transaction
    ↓
Update Payment
6. Payment Status

pending
processing
success
failed
cancelled
refunded
partially_refunded

The exact statuses may depend on the selected payment gateway.

Training

8. Payment Ownership

Payments must be associated with the correct user.

Example:

Candidate
    ↓
Own Payment
    ✓ Allowed

Candidate
    ↓
Another Candidate's Payment
    ✗ Denied

Recruiters should only access payment records belonging to their authorized company or account.

9. Payment Gateway

Career77 should integrate with a suitable payment gateway based on:

Supported countries
Currency support
Payment methods
Webhook support
Security
Reliability
Transaction fees

The exact payment provider can be selected during implementation.

10. Payment Creation

Payment creation should happen through the backend.

Example:

POST /api/payments/create

Flow:

User
    ↓
Select Product
    ↓
Backend Creates Order
    ↓
Generate Payment Reference
    ↓
Return Payment Details
    ↓
Open Payment Gateway
11. Order and Payment

An order and payment should be treated as separate concepts.

Order
    ↓
Represents Purchase Intent

Payment
    ↓
Represents Payment Transaction

Example:

Training
    ↓
Order Created
    ↓
Payment Created
    ↓
Payment Completed
    ↓
Enrollment Created
12. Payment Webhook

The gateway webhook is responsible for communicating payment events to Career77.

Payment Gateway
    ↓
Webhook
    ↓
POST /api/payments/webhook
    ↓
Verify Webhook
    ↓
Process Event

Webhook requests must be validated securely.

13. Webhook Idempotency

Payment webhooks may be delivered multiple times.

Career77 must prevent duplicate processing.

Example:

Webhook Event
    ↓
Check Event ID
    ↓
Already Processed?
    ├── Yes → Ignore Duplicate
    └── No  → Process

This prevents:

Duplicate Enrollment
Duplicate Subscription
Duplicate Payment Record
14. Payment Success

After successful verification:

Payment Success

15. Training Enrollment

Training access should only be granted after successful payment verification.

Payment
    ↓
Verified Success
    ↓
Enrollment Created
    ↓
Candidate Access Granted

A failed or pending payment must not grant paid training access.

16. Payment Failure

If payment fails:

Payment Failed
    ↓
Update Payment Status
    ↓
Do Not Grant Access
    ↓
Notify User

The user may retry the payment.

17. Payment Cancellation

If the user cancels payment:

Payment Cancelled
    ↓
Update Payment Status
    ↓
No Service Activation

23. Payment Security

Career77 should:

Never store raw card details
Never store CVV
Use gateway-hosted or gateway-secured payment flows
Keep gateway secrets server-side
Verify payments server-side
Validate webhooks
Prevent duplicate processing
Log important payment events
24. Environment Variables

Payment gateway credentials should be stored securely.

Example:

PAYMENT_GATEWAY_KEY
PAYMENT_GATEWAY_SECRET
PAYMENT_WEBHOOK_SECRET

The actual variable names depend on the selected payment provider.

Secrets must never be exposed to client-side code.

25. Payment API

Possible API structure:

POST   /api/payments/create
GET    /api/payments/[id]
GET    /api/payments
POST   /api/payments/webhook
POST   /api/payments/[id]/refund

The final API design may change during implementation.

26. Payment Service

Payment logic should be centralized.

Recommended structure:

lib/
└── payments/
    ├── payment-service.ts
    ├── order-service.ts
    ├── webhook-service.ts
    └── refund-service.ts

Gateway-specific logic should be separated from general payment business logic.

27. Payment and Notification

After payment status changes:

Payment Event
    ↓
Payment Service
    ↓
Notification Service

Possible notifications:

Payment Successful
Payment Failed
Refund Successful
Subscription Activated
Subscription Expiring
28. Payment Audit Logs

Important payment actions should be recorded.

Examples:

Payment Created
Payment Verified
Payment Failed
Payment Refunded
Subscription Activated

Audit logs should include:

Actor
Action
Payment ID
Reference ID
Timestamp

Sensitive payment credentials must never be stored in audit logs.

30. Final Payment Principle

Career77 payments must be:

Secure
    ↓
Server-Verified
    ↓
Idempotent
    ↓
Auditable
    ↓
Reliable

The platform must never grant paid services based solely on frontend payment results.

Payment verification must always be completed server-side before activating training enrollment, subscriptions, or other paid services.
