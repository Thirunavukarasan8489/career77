1. Purpose

This document defines the notification architecture for Career77.

The notification system keeps users informed about important platform events.

Career77 supports notifications for:

Candidate activities
Recruiter activities
Super Admin activities
Applications
Interviews
Recruiter verification
Training
Payments
Subscriptions
Support
System events
2. Notification Channels

Career77 may use:

In-App Notifications
Email Notifications
WhatsApp Deep Links

The primary notification system should be controlled by the backend.

3. Notification Architecture
Platform Event
    ↓
Notification Service
    ↓
Determine Recipient
    ↓
Determine Notification Type
    ↓
Create In-App Notification
    ↓
Send Email if Required
    ↓
WhatsApp Deep Link if Applicable

18. Email Notifications

Transactional emails may be sent for important events.

Examples:

Welcome Email
Email Verification
Password Reset
Recruiter Approval
Recruiter Rejection
Application Updates
Interview Updates
Payment Confirmation
Subscription Updates
19. Email Provider

Career77 should use a transactional email provider.

The exact provider can be selected based on:

Reliability
Delivery rate
Cost
Developer experience
Production requirements

Email credentials must be stored securely in environment variables.

20. WhatsApp Deep Links

Career77 may use WhatsApp deep links for quick communication or quick apply flows.

Example:

Candidate
    ↓
Job Details
    ↓
Quick Apply
    ↓
WhatsApp Deep Link
    ↓
Open WhatsApp

The platform should not treat a WhatsApp deep link as a replacement for secure application tracking where formal application records are required.

21. Notification Preferences

Users may eventually control notification preferences.

Example:

Email Notifications
In-App Notifications
Application Updates
Interview Updates
Training Updates
Marketing Notifications

Critical system notifications may not be disabled.

22. Read and Unread Status

Notifications should support:

Unread
Read

Example:

Notification
    ↓
Unread
    ↓
User Opens
    ↓
Read
23. Mark as Read

Users may mark notifications as read.

Example:

PATCH /api/notifications/[id]/read

Users should only be able to update their own notifications.

24. Mark All as Read

The notification system may support:

PATCH /api/notifications/read-all

This should only affect notifications belonging to the authenticated user.

25. Notification Pagination

Notification lists should use pagination.

Example:

GET /api/notifications?page=1&limit=20

This prevents large notification collections from being loaded at once.

26. Notification Priority

Notifications may have priority levels.

low
normal
high
critical

Critical notifications should be reserved for important platform events.

27. Notification Data

Notifications may include additional data for navigation.

Example:

{
  "type": "application",
  "data": {
    "applicationId": "123",
    "jobId": "456"
  }
}

The frontend can use this information to navigate to the relevant page.

28. Notification Service

Notification logic should be centralized.

Recommended structure:

lib/
└── notifications/
    ├── notification-service.ts
    ├── email-service.ts
    └── templates/

The exact structure may change based on implementation requirements.

29. Notification Event Flow
Business Event
    ↓
Application Service
    ↓
Notification Service
    ├── In-App Notification
    ├── Email
    └── WhatsApp Link
30. Notification Security

Notifications must respect user permissions.

A user should never receive notifications containing another user's private data.

Example:

Candidate A
    ↓
Candidate A Application
    ↓
Candidate A Notification

The notification must not expose private information belonging to another candidate.

31. Notification Failure

Notification failures should not necessarily fail the main business operation.

Example:

Application Created
    ↓
Application Saved Successfully
    ↓
Email Failed

The application should remain successfully created.

The notification failure should be logged and handled separately.

32. Retry Strategy

Important notification delivery failures may be retried.

Example:

Notification Failed
    ↓
Retry
    ↓
Retry Failed
    ↓
Log Failure

The exact retry strategy depends on the selected email infrastructure.

33. Notification Logging

Important notification events may be logged.

Examples:

Email Sent
Email Failed
Notification Created
Notification Read

Sensitive information should not be stored unnecessarily in logs.

35. Final Principle

Career77 notifications should provide timely and relevant communication across the platform.

The notification system should be:

Reliable
    ↓
Secure
    ↓
Relevant
    ↓
User-Friendly
    ↓
Scalable

All notifications must respect user roles, resource ownership, privacy, and authorization rules.
