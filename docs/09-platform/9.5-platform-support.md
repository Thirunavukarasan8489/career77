1. Purpose

This document defines the support architecture for Career77.

The support system allows users to report issues, ask questions, and request assistance.

Career77 support users include:

Candidate
Recruiter
Super Admin
2. Support Architecture
User
    ↓
Create Support Ticket
    ↓
Ticket Created
    ↓
Support Queue
    ↓
Super Admin Review
    ↓
Reply / Resolve
    ↓
User Notification
3. Support Ticket Types

Users may create tickets for:

Account
Login
Profile
Job Application
Recruiter Verification
Payment
Training
Subscription
Technical Issue
Other
4. Candidate Support

Candidates may create support tickets for:

Account Issues
Login Problems
Application Issues
Interview Issues
Training Issues
Payment Issues
Technical Problems

Candidates should only access their own support tickets.

5. Recruiter Support

Recruiters may create support tickets for:

Account Issues
Company Setup
Recruiter Verification
Job Management
Applicant Management
Subscription
Payment
Technical Problems

Recruiters should only access tickets belonging to themselves or their authorized company.

6. Super Admin Support

Super Admin manages the platform support queue.

Super Admin may:

View Tickets
Assign Tickets
Reply to Tickets
Change Priority
Change Status
Resolve Tickets
Reopen Tickets
7. Support Ticket Lifecycle
Open
    ↓
Assigned
    ↓
In Progress
    ↓
Waiting for User
    ↓
Resolved
    ↓
Closed

A ticket may be reopened if the issue is not resolved.

8. Ticket Status

Recommended statuses:

open
assigned
in_progress
waiting_for_user
resolved
closed
9. Ticket Priority

Tickets may have:

low
normal
high
urgent

Urgent tickets should be reviewed first.

10. Ticket Data

A support ticket may contain:

_id
ticketNumber
userId
companyId
category
subject
description
priority
status
assignedTo
createdAt
updatedAt
resolvedAt
closedAt
11. Ticket Number

Each ticket should have a unique human-readable ticket number.

Example:

CV77-100001
CV77-100002
CV77-100003

The internal MongoDB _id should remain separate from the public ticket number.

12. Support Conversation

Each ticket may contain a conversation.

Support Ticket
    ↓
Messages
    ├── User
    ├── Super Admin
    └── Support Team

Users should only see messages belonging to their own tickets.

13. Ticket Creation Flow
User
    ↓
Open Support
    ↓
Select Category
    ↓
Enter Subject
    ↓
Enter Description
    ↓
Submit
    ↓
Create Ticket
    ↓
Generate Ticket Number
    ↓
Notify Support
14. Ticket Assignment

Super Admin may assign a ticket to a support administrator.

New Ticket
    ↓
Support Queue
    ↓
Assign Support User
    ↓
In Progress

The assignment process may be expanded when a dedicated support team is introduced.

15. Ticket Reply

Flow:

Support User
    ↓
Open Ticket
    ↓
Reply
    ↓
Save Message
    ↓
Notify User

The user can then reply to the ticket.

16. Waiting for User

When support requires additional information:

Support
    ↓
Request More Information
    ↓
Status = Waiting for User

When the user replies:

User Reply
    ↓
Status = In Progress
17. Ticket Resolution

When the issue is resolved:

Support
    ↓
Resolve Ticket
    ↓
Status = Resolved
    ↓
Notify User

The user may confirm the resolution or request reopening.

18. Ticket Closing

A resolved ticket may eventually become:

Resolved
    ↓
Closed

Closed tickets should remain available as historical records according to the platform's retention policy.

19. Reopen Ticket

If the issue is not resolved:

Resolved Ticket
    ↓
User Requests Reopen
    ↓
Ticket Reopened
    ↓
In Progress

The ticket should maintain its existing history.

Ticket Resolved

Super Admin may receive notifications for:

Urgent Ticket

Important support events may also trigger email notifications.

Examples:

Ticket Created
New Support Reply
Ticket Resolved
Ticket Reopened

Email notifications should contain a link to the relevant support ticket.

23. Support Ticket Privacy

Support tickets may contain sensitive information.

Therefore:

Candidate
    ↓
Own Tickets Only
Recruiter
    ↓
Own / Authorized Company Tickets
Super Admin
    ↓
Authorized Administrative Access

Unauthorized users must not access ticket details.

    ↓

    ↓

25. Attachment Security

Support attachments must follow:

File type validation
File size limits
Authentication
Authorization
Ownership checks

Private attachments should not be publicly accessible without authorization.

Career77 may define support response targets in the future.

Example:

Urgent
    → Highest Priority

High
    → Fast Response

Normal
    → Standard Response

Low
    → Standard Queue

The exact SLA values should be defined separately.

33. Support Security

Career77 support must:

Require authentication.
Enforce ticket ownership.
Restrict Super Admin access.
Validate attachments.
Protect private information.
Prevent unauthorized ticket modification.
Log sensitive administrative actions.
Apply rate limiting to ticket creation.
Prevent spam.
Maintain ticket history.

35. Final Support Principle

Career77 support should provide a structured and transparent way for users to resolve platform issues.

The support system should be:

Simple
    ↓
Secure
    ↓
Trackable
    ↓
Role-Based
    ↓
Auditable
    ↓
Scalable

Every support ticket must follow authentication, authorization, ownership, privacy, and audit rules.
