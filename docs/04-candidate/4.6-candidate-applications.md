1. Purpose

The Candidate Applications feature allows Candidates to manage and track jobs they have applied for.

Candidates can:

View Applications
Track Application Status
View Application Details
Withdraw Applications
View Application History
2. Applications Access

Route:

/candidate/applications

Access requires:

Authenticated
+
Role = Candidate
+
Account Status = Active
3. Application Flow
Candidate
    ↓
Search Jobs
    ↓
View Job
    ↓
Apply
    ↓
Select Resume
    ↓
Submit Application
    ↓
Application Created
    ↓
Track Application
4. Applications List

The Applications page displays all jobs applied for by the Candidate.

Each application may show:

Job Title
Company
Location
Applied Date
Application Status
Last Updated

Example:

Frontend Developer
ABC Technologies

Applied: 20 July 2026
Status: Under Review

Applied

6. Application Details

Candidates can open an application to view details.

Information may include:

Job Title
Company
Job Location
Applied Date
Application Status
Submitted Resume
Application Details
Status History
7. Application Status Timeline

The system should display application progress as a timeline.

Example:

✓ Applied
    20 July 2026

✓ Under Review
    21 July 2026

● Shortlisted
    22 July 2026

○ Interview
    Pending

The timeline helps Candidates understand their application progress.

8. Application History

Candidates should be able to view important application status changes.

Example:

20 July
Application submitted

21 July
Application moved to Under Review

22 July
Candidate shortlisted

Status history should be displayed in chronological order.

9. Withdraw Application

Candidates may withdraw an application if the feature is enabled.

Flow:

Application
    ↓
Withdraw
    ↓
Confirmation
    ↓
Withdraw Application
    ↓
Status = Withdrawn

Confirmation message:

Are you sure you want to withdraw this application?

The system should prevent accidental withdrawal.

10. Duplicate Applications

The system should prevent duplicate applications for the same job.

Example:

Candidate
    ↓
Already Applied
    ↓
Apply Again
    ↓
Application Already Exists

The Apply button should display:

Already Applied

The Candidate can open the existing application instead.

11. Resume Used for Application

The Candidate should be able to see which resume was submitted.

Example:

Submitted Resume

Gokul_Resume_FullStack.pdf

[ View Resume ]

The submitted resume should remain associated with the application.

Updating the Candidate's current resume should not automatically change the resume already submitted with an application.

Status

13. Application Sorting

Candidates may sort applications by:

Newest

14. Empty State

If the Candidate has no applications:

You haven't applied for any jobs yet.

Find your next opportunity on Career77.

Action:

[ Search Jobs ]
15. Loading State

While applications are loading:

Applications
    ↓
Loading
    ↓
Skeleton Application Cards / Table
    ↓
Applications Loaded

The interface should avoid unnecessary layout shifts.

16. Error State

If applications cannot be loaded:

Unable to load your applications.

Please try again.

Action:

[ Retry ]

Internal server errors must not be displayed to Candidates.

17. Notifications

Candidates may receive notifications when an application status changes.

Examples:

Application Submitted
Application Under Review
Application Shortlisted
Interview Scheduled
Application Rejected

Example:

Your application for Frontend Developer
has been shortlisted.
18. Application Data Security

Candidates can only access their own applications.

Candidate A
    ↓
Candidate A Applications

Candidate A must not access:

Candidate B Applications

The backend must verify Candidate ownership for every application request.

19. Recruiter Updates

Recruiters may update application status based on the hiring process.

Example:

Recruiter
    ↓
Review Candidate
    ↓
Update Application Status
    ↓
Candidate Notification
    ↓
Candidate Views Updated Status

Candidates should only see statuses that are approved for Candidate visibility.

20. Responsive Design

The Applications feature must support:

Desktop
Tablet
Mobile

Desktop:

Applications Table

Mobile:

Application Cards

Application details should remain easy to read on all screen sizes.

21. Application Security

The system must:

Require Candidate authentication.
Verify Candidate ownership.
Protect application data.
Prevent unauthorized access.
Validate application requests.
Prevent duplicate applications.
Secure submitted resumes.
Enforce backend authorization.
22. Application Acceptance Criteria

The Applications feature is complete when:

Candidate can view applications.
Candidate can view application details.
Candidate can track application status.
Candidate can view status history.
Candidate can view the submitted resume.
Candidate can withdraw an application where allowed.
Duplicate applications are prevented.
Candidates can filter applications.
Candidates can sort applications.
Application status changes are notified.
Empty states are handled.
Loading states are handled.
Error states are handled.
Candidate data is protected.
Backend authorization is enforced.
Applications are responsive.
Applications are accessible.
