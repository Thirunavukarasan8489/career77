1. Purpose

The Career77 Authentication System provides secure identity management for all protected platform functionality.

The authentication system must support three primary platform roles:

Candidate
Recruiter
Super Admin

The primary objectives are:

Secure user registration.
Secure user login.
User authentication.
Role-based access.
Session management.
Logout.
Password management.
Account recovery.
Email verification where required.
Protected routes.
Authentication state management.
Secure authorization integration.

The authentication flow is:

User
    ↓
Register / Login
    ↓
Authentication
    ↓
Identity Verification
    ↓
Role Identification
    ↓
Authorization
    ↓
Access Platform

3. Authentication vs Authorization

Authentication and authorization are separate responsibilities.

Authentication

Authentication answers:

Who are you?

Example:

User
    ↓
Login
    ↓
Verify Credentials
    ↓
Authenticated
Authorization

Authorization answers:

What are you allowed to access?

Example:

Authenticated User
    ↓
Role = Recruiter
    ↓
Access Recruiter Portal

The system must always authenticate the user before evaluating protected role permissions.

    ↓

    ↓

    ↓

16. Authentication Session

After successful login, the system must create an authenticated session.

The session should represent:

User Identity
Role
Authentication Status
Session Expiration

The exact session implementation must follow the approved Authentication Architecture.

Possible implementation approaches include:

Secure HTTP-only Cookies

or:

Access Token + Refresh Token

The selected architecture must be implemented consistently across frontend and backend.

17. Secure Cookie Requirements

If cookie-based authentication is used, authentication cookies should use appropriate security attributes.

Expected controls include:

HttpOnly
Secure
SameSite

Cookie configuration must follow the production environment.

The authentication cookie must not be accessible through client-side JavaScript when HttpOnly protection is required.

    ↓

Clear Authentication State

Redirect to Public Page

Logout must invalidate the appropriate server-side session or refresh-token mechanism where applicable.

The client must clear local authentication state.

The reset process must use a secure, time-limited token.

22. Password Reset Token

Password reset tokens must:

Be cryptographically secure.
Expire after a limited period.
Be single-use.
Be invalidated after successful password reset.
Not expose user credentials.
Not be stored in plain form where avoidable.

The system should prevent token reuse.

23. Forgot Password Security

The Forgot Password endpoint must not reveal whether an email exists.

Instead of:

Email does not exist.

Use a generic response:

If an account exists for this email,
you will receive password reset instructions.

This helps prevent account enumeration.

39. Current User

The platform should provide a secure way to retrieve the authenticated user.

Conceptual endpoint:

GET /api/auth/me

The response should contain only required user information.

Example:

{
  "success": true,
  "data": {
    "id": "user-id",
    "name": "Gokul",
    "email": "user@example.com",
    "role": "candidate"
  }
}

Sensitive information must not be returned.

42. Account Enumeration Protection

The authentication system must avoid revealing whether an account exists.

Examples:

Login:

Invalid email or password.

Forgot Password:

If an account exists, reset instructions will be sent.

Registration:

The system must follow the approved policy for duplicate account handling.

43. Rate Limiting

Authentication endpoints must be protected against abuse.

Rate limiting should apply to sensitive operations such as:

Login
Forgot Password
Password Reset
Email Verification
Resend Verification

Repeated failed login attempts should trigger appropriate protections.

The exact thresholds must be defined in the Security specification.

44. Brute Force Protection

The system should protect authentication endpoints against brute-force attacks.

Potential controls:

Rate Limiting
Temporary Lockout
Progressive Delays
IP Monitoring
Suspicious Activity Detection

The implementation must balance security and legitimate user access.

45. CSRF Protection

If cookie-based authentication is used, the platform must implement appropriate CSRF protection.

The exact implementation depends on:

Cookie Configuration
SameSite Policy
Frontend Architecture
Backend Framework

CSRF protection must be considered for all state-changing operations.

46. XSS Protection

Authentication-related interfaces must protect against cross-site scripting.

Requirements:

Validate inputs.
Sanitize user-generated content.
Avoid unsafe HTML rendering.
Apply appropriate security headers.
Never inject untrusted values into HTML.

Authentication tokens and sensitive credentials must not be exposed to client-side scripts unnecessarily.

47. Session Expiration

Authenticated sessions must expire according to the approved security policy.

Conceptual flow:

Active Session
    ↓
Expiration
    ↓
Authentication Invalid
    ↓
User Must Reauthenticate

The exact expiration rules must be defined in the Authentication Architecture.

Expirable.
