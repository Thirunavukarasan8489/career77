1. Purpose

This document defines accessibility standards for the Career77 platform.

Career77 should provide an accessible experience for all users, including users with:

Visual disabilities
Hearing disabilities
Motor disabilities
Cognitive disabilities
Temporary impairments

Accessibility applies to:

Public pages
Candidate portal
Recruiter portal
Super Admin portal
Authentication
Forms
Navigation
Tables
Modals
Notifications
Payments
Training
2. Accessibility Goal

Career77 should follow WCAG accessibility principles.

The main principles are:

Perceivable
    ↓
Operable
    ↓
Understandable
    ↓
Robust

The application should aim for a strong level of accessibility across all major user flows.

3. Keyboard Navigation

All interactive elements must be usable with a keyboard.

Users should be able to navigate:

Links
Buttons
Inputs
Forms
Menus
Dropdowns
Dialogs
Tabs
Tables

Avoid keyboard traps.

4. Visible Focus

Keyboard users must always have a visible focus indicator.

Focus states should be clearly visible for:

Buttons
Links
Inputs
Selects
Checkboxes
Tabs
Navigation items

Do not remove default focus styles without providing an accessible replacement.

5. Semantic HTML

Use semantic HTML wherever possible.

Recommended elements:

<header>
<nav>
<main>
<aside>
<section>
<article>
<footer>
<button>
<form>
<label>

Avoid using generic <div> elements when a semantic element is more appropriate.

6. Heading Structure

Use headings in a logical hierarchy.

Example:

H1
    ↓
H2
    ↓
H3

Each page should have a clear primary heading.

Avoid skipping heading levels without a valid reason.

7. Page Titles

Every page should have a meaningful title.

Examples:

Career77 — Find Jobs

Career77 — Candidate Dashboard

Career77 — Recruiter Dashboard

Career77 — Manage Applicants

Page titles should help users understand their current location.

8. Form Labels

Every form field must have a clear label.

Example:

Email Address
[________________]

Do not rely only on placeholder text as a label.

9. Form Instructions

Complex forms should provide clear instructions.

Examples:

Upload your resume in PDF format.

Password must contain at least 8 characters.

Instructions should appear before or near the relevant field.

10. Required Fields

Required fields should be clearly identified.

Example:

Full Name *
Email Address *
Phone Number *

Do not communicate required status using color alone.

11. Form Validation

Validation messages should be:

Clear
Specific
Actionable

Example:

Invalid email address.

Please enter a valid email address.

Avoid technical messages such as:

Validation Error 422
12. Error Identification

Errors should be communicated using:

Text
+
Icon Where Appropriate
+
Color

Do not use color alone to identify errors.

13. Error Recovery

When an error occurs, users should understand how to recover.

Example:

Unable to load applications.

[Try Again]

For form errors:

Please correct the highlighted fields.
14. Color Contrast

Text and important UI elements should maintain sufficient contrast.

Pay special attention to:

Body text
Small text
Placeholder text
Buttons
Links
Status badges
Form errors

Use the color system defined in design-color-system.md.

15. Color Is Not the Only Indicator

Do not rely only on color to communicate meaning.

Use:

Color
+
Text
+
Icon

Example:

✓ Approved
! Pending
× Rejected
16. Link Accessibility

Links should have meaningful text.

Prefer:

View Job Details
View Candidate Profile
View Company

Avoid:

Click Here
Read More

when the destination is unclear.

17. Button Accessibility

Buttons should describe the action.

Good:

Apply Now
Save Job
Create Job
Delete User
Approve Recruiter

Avoid unclear labels:

Click
Submit
Go

when more specific wording is possible.

18. Icon Buttons

Icon-only buttons must provide accessible labels.

Example:

[🔔]

The button should have an accessible name such as:

Notifications

Use tooltips as additional support, not as the only accessible label.

19. Images

Images should have appropriate alternative text.

Examples:

Company Logo
Candidate Profile Photo
Training Illustration

Decorative images should use appropriate empty alternative text where applicable.

20. Company Logos

Company logos should have meaningful alternative text.

Example:

Alt:
"Acme Technologies company logo"

If the logo is purely decorative, it should not create unnecessary screen reader output.

21. Profile Images

Candidate and recruiter profile images should use appropriate alternative text.

Example:

"Gokulnath S profile photo"

Avoid unnecessary descriptions when the image is only decorative.

22. Decorative Icons

Decorative icons should not be unnecessarily announced by screen readers.

Use appropriate accessibility attributes where required.

23. Navigation Accessibility

Navigation should provide:

Clear labels
Logical order
Keyboard access
Visible active state
Accessible mobile menu

Users should always understand where they are.

24. Active Navigation

Active navigation should use more than color alone.

Recommended:

Active Background
+
Active Text
+
Optional Icon Indicator

This improves accessibility and usability.

25. Mobile Navigation

Mobile menus should support:

Keyboard access
Focus management
Clear close action
Screen reader labels

When a menu opens, focus should be handled appropriately.

26. Modal Accessibility

Modals should:

Have a clear title
Have an accessible description where needed
Manage focus correctly
Allow keyboard closing
Prevent background interaction when appropriate

Example:

Confirm Delete

Are you sure you want to delete this job?

[Cancel] [Delete]
27. Drawer Accessibility

Drawers should follow similar accessibility rules as dialogs.

Users should be able to:

Open the drawer
Navigate within it
Close it
Return focus to the triggering element
28. Tables

Tables should have clear:

Column headers
Row relationships
Table labels
Sorting information where applicable

Complex data tables should remain understandable for keyboard and screen reader users.

29. Data Table Actions

Row actions should have meaningful accessible labels.

Avoid:

[⋮]

without context.

Use accessible labels such as:

More actions for John Doe
30. Status Information

Status should be clearly communicated.

Examples:

Application Status: Shortlisted

Recruiter Verification: Pending

Payment Status: Successful

Do not rely only on colored badges.

31. Loading States

Loading states should provide accessible feedback.

Example:

Loading applications...

Screen readers should receive appropriate status updates where necessary.

32. Skeleton Loading

Skeletons should not create confusing screen reader output.

The accessible experience should communicate that content is loading without announcing unnecessary decorative elements.

33. Toast Notifications

Important toast messages should be accessible.

Examples:

Application submitted successfully.

Job saved successfully.

Payment failed. Please try again.

Use appropriate live-region behavior where necessary.

34. Notifications

Notifications should clearly communicate:

Notification Type
Message
Time
Action
Read / Unread Status

Unread status should not rely only on color.

35. Keyboard Focus Order

Focus order should follow the visual and logical order of the page.

Example:

Header
    ↓
Navigation
    ↓
Main Content
    ↓
Actions

Avoid unexpected focus jumps.

36. Skip Navigation

Long pages should provide a mechanism to skip repeated navigation.

Example:

Skip to Main Content

This is especially useful for keyboard users.

37. Text Readability

Text should be:

Clear
Legible
Properly sized
Adequately spaced

Avoid very small text for important information.

38. Text Scaling

The interface should remain usable when text is enlarged.

Important content should not:

Overlap
Disappear
Become inaccessible
Break the layout
39. Responsive Accessibility

Accessibility must be maintained across:

Mobile
Tablet
Desktop
Large Desktop

A feature that is accessible on desktop must remain accessible on mobile.

40. Touch Accessibility

Mobile controls should have sufficient touch target sizes.

Pay special attention to:

Buttons
Links
Icon buttons
Checkboxes
Navigation
Filters

Avoid placing interactive controls too close together.

41. Authentication Accessibility

Authentication pages should provide:

Clear labels
Password visibility option
Error messages
Keyboard navigation
Focus states
Accessible recovery flows

Examples:

Login
Register
Forgot Password
Reset Password
Email Verification
42. Candidate Accessibility

Candidate workflows should support accessible:

Job Search
Job Filters
Job Details
Application
Resume Upload
Interview Scheduling
Training Enrollment
Payment
43. Recruiter Accessibility

Recruiter workflows should support accessible:

Job Creation
Applicant Management
Hiring Pipeline
Candidate Review
Interview Scheduling
Analytics

Complex enterprise workflows should remain keyboard accessible.

44. Super Admin Accessibility

Super Admin workflows should support accessible:

User Management
Company Management
Recruiter Verification
Job Management
Billing
Subscriptions
Support
Analytics

Data-heavy interfaces should remain navigable without relying on a mouse.

45. AI Accessibility

AI features should be clearly identified.

Examples:

AI Match Score
AI Recommendation
AI Resume Feedback
AI Hiring Insight

AI-generated information should be presented in a readable and understandable way.

46. Paid Training Accessibility

Training features should support accessible:

Training Discovery
Course Details
Enrollment
Payment
Learning Progress
Certificates

Important training information should not depend only on visual design.

47. Payment Accessibility

Payment workflows should provide:

Clear labels
Clear pricing
Error messages
Payment status
Confirmation feedback

Example:

Training Price
Discount
Tax
Total
Payment Status
48. Motion and Animation

Animations should be subtle and purposeful.

Avoid excessive:

Flashing
Rapid movement
Auto-playing animations
Distracting transitions

Where appropriate, respect user preferences for reduced motion.

49. Reduced Motion

The application should support reduced-motion preferences where possible.

Users who prefer reduced motion should not be forced to experience unnecessary animations.

50. Auto-Playing Content

Avoid automatic audio or video playback.

If media is required:

Provide user controls
Provide captions where applicable
Allow pause and stop
Avoid unexpected sound
51. Accessibility Testing

Accessibility should be tested using:

Keyboard Testing
Screen Reader Testing
Color Contrast Testing
Responsive Testing
Automated Accessibility Testing
52. Accessibility Testing Checklist

Verify:

✓ Keyboard Navigation Works
✓ Focus Is Visible
✓ Forms Have Labels
✓ Errors Are Clear
✓ Buttons Are Descriptive
✓ Images Have Appropriate Alt Text
✓ Color Contrast Is Sufficient
✓ Color Is Not the Only Indicator
✓ Modals Manage Focus
✓ Tables Have Clear Headers
✓ Mobile Navigation Is Accessible
✓ Touch Targets Are Usable
53. Automated Testing

Use accessibility testing tools during development.

Possible tools:

axe
Lighthouse
eslint-plugin-jsx-a11y

Automated tools should support manual accessibility testing, not replace it.

54. Accessibility in Development

Accessibility should be considered during:

Design
    ↓
Component Development
    ↓
Feature Development
    ↓
Testing
    ↓
Production Review

Accessibility should not be treated only as a final testing step.

55. Accessibility and Design System

All reusable components should include accessibility considerations.

Components such as:

Button
Input
Select
Dialog
Drawer
Tabs
Dropdown
Table
Toast

should have consistent accessible behavior.

56. Accessibility and Future Dark Theme

Career77 currently supports Light Theme only.

When Dark Theme is introduced later, accessibility requirements must continue to apply.

The Dark Theme should be reviewed for:

Color contrast
Focus visibility
Status colors
Disabled states
Text readability
Form states

Dark Theme should not reduce accessibility.

57. Final Accessibility Principle

Career77 should follow:

Accessible by Design
    ↓
Keyboard Friendly
    ↓
Screen Reader Friendly
    ↓
Color Accessible
    ↓
Touch Friendly
    ↓
Responsive
    ↓
Inclusive

Accessibility should be treated as a core product requirement across the Candidate, Recruiter, Super Admin, and public experiences, ensuring users can discover, apply, recruit, manage, pay, learn, and complete important workflows regardless of how they interact with the platform.
