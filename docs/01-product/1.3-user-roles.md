1. Purpose

This document defines the approved user roles and responsibilities within the Career77 platform.

Career77 contains exactly three primary application roles:

Candidate
Recruiter
Super Admin

These roles represent three distinct responsibilities within the platform:

Candidate
    ↓
Finds and applies for jobs

Recruiter
    ↓
Manages jobs and hiring

Super Admin
    ↓
Manages and governs the platform

The role model must remain simple and clearly separated.

No additional top-level user roles should be introduced without explicit product approval.

2. Role Architecture

The Career77 role hierarchy is:

Career77
│
├── Candidate
│
├── Recruiter
│
└── Super Admin

Each role has a separate application experience.

Candidate
    ↓
Candidate Portal

Recruiter
    ↓
Recruiter Portal

Super Admin
    ↓
Super Admin Console

Public visitors are not considered authenticated platform roles.

They interact with the Public / Marketing experience until they register or request access.

24. Candidate and Recruiter Boundary

Candidate and Recruiter experiences must remain separate.

A Candidate:

CAN
    ├── Search Jobs
    ├── Apply
    ├── Manage Profile
    └── Track Applications

CANNOT
    ├── Create Recruiter Jobs
    ├── Manage Applicants
    ├── Manage Hiring Pipelines
    └── Access Admin Console

A Recruiter:

CAN
    ├── Create Jobs
    ├── Manage Applicants
    ├── Manage Hiring Pipelines
    └── Manage Interviews

CANNOT
    ├── Apply as a Candidate through Recruiter Permissions
    ├── Modify Candidate-Owned Profile Data
    └── Access Super Admin Console

If the product later supports one person holding multiple roles, that must be explicitly defined as a separate authentication and authorization requirement.

The current product assumes role-specific access boundaries.

26. Authentication vs Authorization

Career77 must clearly separate:

Authentication
    ↓
Who is the user?

Authorization
    ↓
What can the user access?

Example:

A Recruiter may successfully authenticate.

However, successful authentication does not automatically allow access to every company resource.

The backend must verify:

Authenticated User
    ↓
User Role
    ↓
Company Association
    ↓
Resource Ownership
    ↓
Permission
    ↓
Allow / Deny
27. Role-Based Access Control

The platform must implement role-based access control.

The base roles are:

ROLE_CANDIDATE
ROLE_RECRUITER
ROLE_SUPER_ADMIN

Role permissions must be enforced server-side.

Example:

Candidate
    → Candidate APIs

Recruiter
    → Recruiter APIs

Super Admin
    → Admin APIs

The frontend must hide unauthorized navigation items, but frontend visibility is not a security mechanism.

Backend authorization is mandatory.

28. Resource-Level Authorization

Role authorization alone may not be sufficient.

Career77 must also enforce resource-level authorization.

Example:

Recruiter A
    ↓
Company A
    ↓
Job A
    ↓
Applicants for Job A

Recruiter A must not access:

Company B
Job B
Applicants for Job B

unless explicitly authorized.

Similarly:

Candidate A
    ↓
Application A

Candidate A must not access:

Candidate B
Application B
Resume B

unless a specific approved workflow grants access.

31. Role Security Requirements

The platform must:

Validate authentication on protected routes.
Validate role authorization.
Validate resource ownership.
Protect sensitive APIs.
Prevent horizontal privilege escalation.
Prevent vertical privilege escalation.
Validate all server-side permissions.
Protect private files.
Protect candidate information.
Protect recruiter information.
Protect administrative operations.
Log sensitive administrative operations where required.

Super Admin operations should be auditable.

32. Role-Based Navigation Rules

Navigation must dynamically reflect the authenticated user's role.

IF role = Candidate
    → Show Candidate Navigation

IF role = Recruiter
    → Show Recruiter Navigation

IF role = Super Admin
    → Show Admin Navigation

Unauthorized navigation items must not be displayed.

However:

Hidden Navigation ≠ Security

Every protected API and server-side action must independently enforce authorization.

33. Role Acceptance Criteria

The role architecture is complete when:

Exactly three primary application roles exist.
Candidate role is implemented.
Recruiter role is implemented.
Super Admin role is implemented.
Candidate registration is self-service.
Recruiter access uses Request Access.
Recruiter approval is managed by Super Admin.
Super Admin registration is not public.
Candidate data ownership is enforced.
Recruiter company ownership is enforced.
Super Admin platform access is protected.
Role-based navigation is implemented.
Role-based API authorization is implemented.
Resource-level authorization is implemented.
Unauthorized direct URL access is blocked.
Cross-role data access is restricted.
Sensitive administrative actions are auditable where required.
No additional top-level roles are introduced without approval.
