1. Purpose

This document defines the security standards for the Career77 platform.

Security must protect:

Candidate data
Recruiter data
Company data
Super Admin data
Resumes
Certificates
Profile images
Payment information
Authentication sessions
Support conversations
System configuration
2. Security Principles

Career77 follows these principles:

Authentication
    ↓
Authorization
    ↓
Input Validation
    ↓
Secure Data Handling
    ↓
Least Privilege
    ↓
Audit Logging
    ↓
Monitoring

Security must be implemented on both frontend and backend.

3. User Roles

Career77 has three primary roles:

Candidate
Recruiter
Super Admin

Every protected feature must verify:

User Authentication
+
User Role
+
Required Permission
4. Authentication

Career77 uses NextAuth.js for authentication.

Authentication must be required for protected areas:

/candidate/*
/recruiter/*
/admin/*

Public pages should only expose information intended for public access.

5. Authentication Flow
User
    ↓
Login
    ↓
Validate Credentials
    ↓
Create Secure Session
    ↓
Authenticated User
    ↓
Access Authorized Portal

Invalid credentials must not reveal sensitive account information.

6. Password Security

Passwords must:

Never be stored in plain text.
Never be logged.
Never be returned through APIs.
Use secure hashing.
Follow the platform password policy.

Password reset links should be:

Time-limited.
Single-use.
Securely generated.
7. Session Security

Sessions must be securely managed.

Consider:

Session Expiration
Secure Cookies
HttpOnly Cookies
SameSite Policy
Session Revocation
Logout

Sensitive authentication information must not be stored insecurely in browser storage.

8. Authorization

Authentication answers:

Who is the User?

Authorization answers:

What can the User access?

Every protected API must perform authorization checks.

Example:

Candidate
    ↓
Request Admin API
    ↓
Authorization Check
    ↓
403 Forbidden
9. Role-Based Access Control

Career77 uses role-based access control.

Candidate
    ├── Manage Own Profile
    ├── Search Jobs
    ├── Apply for Jobs
    └── Manage Own Applications

Recruiter
    ├── Manage Own Company
    ├── Manage Own Jobs
    ├── View Applicants
    └── Manage Hiring Pipeline

Super Admin
    ├── Manage Users
    ├── Verify Recruiters
    ├── Manage Companies
    ├── Manage Jobs
    └── Manage Platform
10. Ownership Checks

Role checks alone are not always sufficient.

The system must also verify resource ownership.

Example:

Recruiter A
    ↓
Request Recruiter B's Job
    ↓
Ownership Check
    ↓
Access Denied

A Recruiter should only manage resources they are authorized to manage.

11. Server-Side Authorization

Security checks must be performed server-side.

Do not rely only on:

Hidden buttons
Frontend route guards
Client-side conditions

The backend must independently verify:

Authentication
Role
Permission
Resource Ownership
12. Input Validation

All external input must be validated.

Validate:

Request Body
Query Parameters
Route Parameters
Form Data
File Uploads
Headers Where Required

Never trust client-side validation.

13. Input Sanitization

User-provided data should be safely handled before storage and rendering.

Examples:

Candidate Name
Job Description
Company Description
Support Messages
Training Content
CMS Content

Prevent malicious content from being executed in the browser.

14. XSS Protection

Career77 must protect against Cross-Site Scripting.

Avoid rendering untrusted HTML directly.

If HTML content is required:

User Input
    ↓
Sanitize
    ↓
Validate
    ↓
Render

CMS and rich-text content require additional sanitization.

15. Injection Protection

Database queries must safely handle user input.

Use Mongoose and validated query parameters.

Avoid constructing unsafe queries directly from raw user input.

Validate:

Search Parameters
IDs
Filters
Sort Values
Pagination Values
16. API Security

All APIs should implement appropriate:

Authentication
Authorization
Validation
Rate Limiting
Error Handling
Logging

Public APIs should expose only required data.

17. Rate Limiting

Rate limiting should be applied to sensitive or high-volume endpoints.

Examples:

Login
Password Reset
Registration
Email Verification
Search
Support Requests
API Requests

The exact limits should be defined based on usage patterns and infrastructure.

18. Brute-Force Protection

Authentication endpoints should protect against repeated failed attempts.

Example:

Failed Login
    ↓
Attempt Limit
    ↓
Temporary Lockout
    ↓
Wait Period
    ↓
Retry

Security events should be monitored.

19. CSRF Protection

State-changing operations must be protected against Cross-Site Request Forgery where applicable.

Examples:

Create Job
Update Profile
Delete Job
Change Password
Payment Actions
Admin Configuration Changes

Use secure authentication and request protection mechanisms appropriate to the Next.js and NextAuth.js architecture.

20. Secure Cookies

Authentication cookies should use appropriate security attributes.

Recommended considerations:

HttpOnly
Secure
SameSite
Appropriate Expiration

Cookie configuration should follow the authentication architecture.

21. Secret Management

Sensitive credentials must never be committed to Git.

Examples:

MONGODB_URI
NEXTAUTH_SECRET
CLOUDINARY_API_SECRET
EMAIL_PROVIDER_API_KEY
PAYMENT_SECRET

Use:

Environment Variables
+
Secure Secret Management
22. Environment Separation

Use separate configurations for:

Development
Staging
Production

Production secrets must never be reused in development environments.

23. Sensitive Data

Protect sensitive information such as:

Passwords
Authentication Tokens
Payment Credentials
API Keys
Private Files
Personal Information

Only collect and store information required for Career77 functionality.

24. Personal Data

Candidate and Recruiter data should be treated as sensitive.

Examples:

Name
Email
Phone
Resume
Profile
Employment History
Education
Company Information

Access should be limited according to role and business requirements.

25. Resume Security

Resumes are stored using Cloudinary.

Resume access must follow authorization rules.

Example:

Candidate
    ↓
Own Resume
    ↓
Allowed

Unauthorized User
    ↓
Private Resume
    ↓
Denied

Public resume URLs should not be exposed unless explicitly required.

26. Certificate Security

Training certificates should be securely stored.

Only authorized Users should access private certificates.

If certificates are publicly verifiable, Career77 should use a controlled verification mechanism.

27. Profile Image Security

Profile images should:

Validate file types.
Validate file size.
Use secure upload flows.
Prevent malicious file uploads.
28. File Upload Security

Before uploading:

Authenticate User
    ↓
Check Permission
    ↓
Validate File Type
    ↓
Validate File Size
    ↓
Upload to Cloudinary
    ↓
Store File Reference

Do not trust the file extension alone.

29. Cloudinary Security

Cloudinary credentials must remain server-side.

Never expose:

CLOUDINARY_API_SECRET

to the browser.

Use secure upload strategies and appropriate access controls.

30. Payment Security

Career77 payment integrations must never store raw card information.

Payment processing should be handled by the selected payment provider.

The application should store only required transaction information.

Examples:

Transaction ID
Payment Status
Amount
Currency
Payment Date
Reference ID
31. Payment Verification

Payment status must be verified server-side.

Do not trust:

Client-Side Success Message

Recommended flow:

Payment
    ↓
Payment Provider
    ↓
Server Verification / Webhook
    ↓
Verify Transaction
    ↓
Update Database
32. Webhook Security

Payment webhooks should:

Verify provider signatures.
Validate payloads.
Prevent duplicate processing.
Handle retries safely.
Log important events.

Example:

Webhook
    ↓
Verify Signature
    ↓
Check Transaction
    ↓
Check Duplicate
    ↓
Update Payment
33. Duplicate Payment Protection

Payment processing should be idempotent.

If the same payment event is received multiple times:

First Event
    ↓
Payment Processed

Duplicate Event
    ↓
Already Processed
    ↓
Ignore Safely
34. Email Security

Transactional email systems should protect:

API Keys
SMTP Credentials
User Information

Email templates should avoid exposing sensitive information unnecessarily.

35. WhatsApp Security

WhatsApp Quick Apply links should only include information required for the intended flow.

Avoid placing sensitive personal information directly in URLs.

Example:

Candidate Private Data
    ↓
Do Not Include in wa.me URL
36. Error Messages

Error messages should be helpful without exposing internal information.

Avoid:

MongoDB connection string:
mongodb://user:password@...

Use:

Something went wrong.
Please try again.

Technical details should be logged securely on the server.

37. Logging Security

Logs must not contain:

Passwords
Access Tokens
API Secrets
Payment Credentials
Private Keys
Sensitive Personal Data

Use structured logging where possible.

38. Audit Logs

Important security and administrative actions should be recorded.

Examples:

Login
Logout
Failed Login
Password Reset
Role Change
Recruiter Approval
Company Verification
Job Approval
Payment Status Change
System Setting Change

Audit records should include:

User
Action
Resource
Timestamp
Result
39. Admin Security

Super Admin accounts require stronger security controls.

Recommended:

Strong Password
Two-Factor Authentication
Secure Sessions
Rate Limiting
Audit Logs
Restricted Access

Admin actions should be carefully audited.

40. Recruiter Verification Security

Recruiter access must follow the verification workflow.

Recruiter Registration
    ↓
Details Validation
    ↓
Company Setup
    ↓
Super Admin Review
    ↓
Approved
    ↓
Welcome Email
    ↓
Recruiter Login

Rejected Recruiters should not receive full Recruiter Portal access.

41. Rejection and Reapproval

If a Recruiter is rejected:

Super Admin Rejects
    ↓
Recruiter Notified
    ↓
Recruiter Requests Review
    ↓
Super Admin Reviews
    ↓
Approve / Reject

Every decision should be recorded in Audit Logs.

42. Database Security

MongoDB access must be restricted.

Use:

Secure Credentials
Network Restrictions
Least Privilege
Encrypted Connections
Backups
Monitoring

The application should not use unnecessary database privileges.

43. Database Backups

Production database backups should be configured.

Backups should be:

Automated where possible.
Securely stored.
Access controlled.
Tested for restoration.

A backup that cannot be restored should not be considered reliable.

44. Dependency Security

Dependencies should be regularly reviewed.

Process:

Dependency Update
    ↓
Security Review
    ↓
Testing
    ↓
Build
    ↓
Deployment

Remove unnecessary dependencies.

45. Security Updates

Security vulnerabilities should be prioritized.

Critical
    ↓
Immediate Action

High
    ↓
Priority Fix

Medium
    ↓
Scheduled Fix

Low
    ↓
Monitor

Critical vulnerabilities should not be ignored.

46. Vercel Security

Career77 is hosted on Vercel.

Production deployments should use:

Secure environment variables.
Protected production configuration.
Preview environments for testing.
Deployment monitoring.

Production secrets should not be exposed in client-side bundles.

47. Security Headers

The application should use appropriate security headers where applicable.

Examples:

Content-Security-Policy
X-Content-Type-Options
Referrer-Policy
Permissions-Policy

Header configuration should be tested to avoid breaking legitimate application functionality.

48. HTTPS

Production Career77 traffic must use HTTPS.

Sensitive communication must not be transmitted over insecure HTTP.

49. Data Encryption

Sensitive data should be protected:

In Transit
    ↓
HTTPS / TLS

At Rest
    ↓
Secure Database / Storage

Encryption requirements should follow the sensitivity of the data.

50. Security Testing

Security testing should include:

Authentication Testing
Authorization Testing
RBAC Testing
Input Validation
XSS Testing
Injection Testing
File Upload Testing
Rate Limit Testing
Session Testing
API Security Testing

Critical security issues must be resolved before production release.

51. Incident Response

If a security incident occurs:

Detect
    ↓
Contain
    ↓
Investigate
    ↓
Fix
    ↓
Recover
    ↓
Monitor
    ↓
Document

Security incidents should be recorded and reviewed.

52. Security Monitoring

Monitor:

Failed Login Attempts
Unusual API Activity
Admin Actions
Payment Failures
File Upload Abuse
Security Errors

Suspicious activity should trigger investigation.

53. Principle of Least Privilege

Every User, service, and system component should receive only the permissions required.

Example:

Candidate
    ↓
Candidate Permissions

Recruiter
    ↓
Recruiter Permissions

Super Admin
    ↓
Administrative Permissions

Avoid giving unnecessary access.

54. Security Checklist

Before production release, verify:

Authentication works correctly.
Authorization is enforced server-side.
RBAC is implemented.
Resource ownership is checked.
Passwords are securely handled.
Sessions are secure.
APIs validate input.
Rate limiting is configured where required.
File uploads are validated.
Cloudinary credentials are protected.
Payment webhooks are verified.
Sensitive data is protected.
Secrets are not committed to Git.
HTTPS is enabled.
Security headers are configured where appropriate.
Audit Logs are enabled.
Database access is restricted.
Backups are configured.
Dependencies are reviewed.
Security tests are completed.
Production monitoring is active.

User Request
    ↓
HTTPS
    ↓
Authentication
    ↓
Session Validation
    ↓
Role Check
    ↓
Permission Check
    ↓
Resource Ownership Check
    ↓
Input Validation
    ↓
Business Logic
    ↓
Database / Cloudinary / External Service
    ↓
Secure Response
    ↓
Audit Log Where Required
    ↓
Monitoring

Career77 security standards ensure that Candidate, Recruiter, and Super Admin data and workflows are protected through secure authentication, server-side authorization, RBAC, input validation, rate limiting, secure file storage, payment verification, secret management, audit logging, monitoring, and controlled production operations.
