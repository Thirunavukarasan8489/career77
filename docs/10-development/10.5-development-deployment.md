1. Purpose

This document defines the deployment standards for the Career77 platform.

The deployment process should provide:

Reliable releases
Secure production configuration
Environment separation
Automated validation
Easy rollback
Minimal downtime
Monitoring after deployment
2. Deployment Architecture

Career77 uses:

Frontend / Application:
Next.js 15+

Database:
MongoDB

ODM:
Mongoose

File Storage:
Cloudinary

Authentication:
NextAuth.js

Hosting:
Vercel

Email:
Transactional Email Provider

Quick Apply:
WhatsApp Deep Link
3. Deployment Environments

Career77 should use separate environments.

Development
    ↓
Local Developer Environment

Preview / Staging
    ↓
Testing and Review

Production
    ↓
Live Career77 Platform

Each environment should have separate configuration and secrets.

4. Development Environment

Developers work locally using:

Next.js
MongoDB
Mongoose
Cloudinary
NextAuth.js
Tailwind CSS

Local environment variables should be stored in:

.env.local

Do not commit .env.local to Git.

5. Preview Environment

Every major feature should be tested in a Preview environment before production.

Preview testing should include:

Authentication
Authorization
Candidate Flow
Recruiter Flow
Admin Flow
Job Search
Applications
Payments
File Uploads
Notifications
6. Production Environment

Production is the live Career77 application.

Production should use:

Production Database
Production Cloudinary
Production Authentication Secrets
Production Email Provider
Production Payment Configuration

Never use development credentials in production.

Career77 is hosted on Vercel.

Deployment should use:

Git Repository
    ↓
Vercel

Build
    ↓
Deploy
    ↓
Production

Vercel should manage application deployments and Preview environments.

12. Environment Variables

Configure environment variables separately for:

Development
Preview
Production

Example:

MONGODB_URI
NEXTAUTH_SECRET
NEXTAUTH_URL
CLOUDINARY_CLOUD_NAME
CLOUDINARY_API_KEY
CLOUDINARY_API_SECRET
EMAIL_PROVIDER_API_KEY

Payment provider variables should also be configured where applicable.

13. Secret Management

Secrets must:

Never be committed to Git.
Never be placed directly in source code.
Never be exposed to Client Components.
Never be included in logs.
Use secure environment configuration.

Only server-side code should access server secrets.

14. Public Environment Variables

Only variables intentionally required by browser code should use the public environment variable prefix.

Do not expose:

Database Credentials
API Secrets
Cloudinary Secret
NextAuth Secret
Payment Secrets
Email Provider Secrets
15. Database Deployment

Production should use a dedicated MongoDB environment.

Recommended:

Development Database
Preview / Staging Database
Production Database

Avoid connecting development code directly to the production database.

16. Database Migration

Before changing production schemas:

Develop Schema Change
    ↓
Test Locally
    ↓
Test in Preview
    ↓
Verify Existing Data
    ↓
Deploy Application
    ↓
Apply Required Data Changes
    ↓
Verify Production

Database changes should be backward compatible where possible.

17. Database Backup

Before major database changes:

Create Backup
    ↓
Perform Change
    ↓
Verify Application
    ↓
Monitor

Production backups should be automated and regularly tested for restoration.

18. Cloudinary Deployment

Production Cloudinary configuration should be separate from development configuration.

Use Cloudinary for:

Profile Images
Resumes
Certificates
Company Logos

Verify upload and access permissions after deployment.

19. Authentication Deployment

After deployment, verify:

Candidate Login
Recruiter Login
Super Admin Login
Logout
Session Management
Protected Routes
Role-Based Access

Authentication secrets must be consistent with the deployed environment.

32. Rollback Strategy

If a deployment causes a critical issue:

Detect Issue
    ↓
Stop Further Changes
    ↓
Identify Deployment
    ↓
Rollback
    ↓
Verify Previous Version
    ↓
Investigate
    ↓
Fix
    ↓
Test
    ↓
Redeploy

Rollback should be performed as quickly as possible for critical failures.

33. Hotfix Process

For critical production issues:

Production Issue
    ↓
Create Hotfix Branch
    ↓
Fix Issue
    ↓
Test
    ↓
Review
    ↓
Deploy
    ↓
Smoke Test

After the hotfix, ensure the fix is merged back into the main development branch.

34. Zero-Downtime Deployment

Where supported by the hosting platform, deployments should minimize downtime.

The application should avoid deployments that cause unnecessary service interruption.

35. Database Compatibility

Application deployments and database changes should be coordinated.

Prefer:

Backward-Compatible Change
    ↓
Deploy New Application
    ↓
Migrate Data
    ↓
Remove Old Logic Later

Avoid breaking the existing production application during schema changes.

36. Maintenance Mode

If planned maintenance is required:

Maintenance Notice
    ↓
Enable Maintenance Mode
    ↓
Perform Maintenance
    ↓
Verify System
    ↓
Disable Maintenance Mode

Maintenance should be communicated to affected users when necessary.

37. Deployment Security

Before production deployment, verify:

HTTPS Enabled

Database Access Restricted
Authentication Working
Authorization Working
Security Headers Configured
