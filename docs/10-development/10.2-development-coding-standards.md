1. Purpose

This document defines the coding standards and development conventions for the Career77 platform.

All developers should follow these standards to maintain:

Clean code
Consistent architecture
Type safety
Reusable components
Secure development
Easy maintenance
Scalable implementation
2. Technology Stack

Career77 uses the following core technologies:

Framework:
Next.js 15+ with App Router

Database:
MongoDB with Mongoose ODM

File Storage:
Cloudinary

Authentication:
NextAuth.js

Search:
MongoDB Text Indexes

Future Search:
MongoDB Atlas Search when required

Styling:
Tailwind CSS

Hosting:
Vercel

Notifications:
Transactional Email Provider

Quick Apply:
WhatsApp Deep Link (wa.me)
3. Next.js Standards

Use:

Next.js 15+
App Router

Prefer the App Router architecture for all new features.

Recommended structure:

app/
├── (public)/
├── (auth)/
├── candidate/
├── recruiter/
├── admin/
├── api/
└── layout.tsx

Use route groups to organize related pages without affecting URL structure.

4. Server and Client Components

Use Server Components by default.

Use Client Components only when client-side functionality is required.

Client Components may be required for:

User interactions
React state
Event handlers
Browser APIs
Client-side hooks

Example:

"use client";

import { useState } from "react";

Avoid adding "use client" unnecessarily.

Keep the Client Component boundary as small as possible.

5. Data Fetching

Prefer server-side data fetching when possible.

Use Server Components for:

Initial page data
Database queries
Server-side rendering
Secure server operations

Use client-side fetching only when required for:

Dynamic interactions
Real-time updates
Client-specific state
User-triggered operations
6. ISR and Caching

Use native Next.js ISR and caching features where appropriate.

ISR is suitable for content that:

Does not change frequently
Can be cached
Benefits from fast delivery

Examples:

Public Job Pages
Company Profiles
Training Course Pages
CMS Content

Dynamic and user-specific data should not be incorrectly cached.

Always consider whether data is:

Static
Revalidated
Dynamic
User-Specific

before selecting a caching strategy.

7. Edge Network

Career77 is hosted on Vercel and should use the platform's Edge Network capabilities where appropriate.

Use edge-compatible solutions only when the feature supports the required runtime.

Avoid using Node.js-only APIs in Edge runtime code.

8. TypeScript

Use TypeScript for application code.

Prefer explicit and meaningful types.

Example:

type Job = {
  id: string;
  title: string;
  location: string;
};

Avoid:

const job: any = {};

Do not use any unless there is a strong technical reason.

Prefer:

unknown

when the type is not known.

9. Type Naming

Use PascalCase for:

Types
Interfaces
Classes
React Components

Example:

type CandidateProfile = {};

interface RecruiterProfile {}

function CandidateDashboard() {}

Use descriptive names.

Avoid unclear names:

type Data = {};
type Info = {};
type Obj = {};
10. Variables and Functions

Use camelCase for:

Variables
Functions
Object properties

Example:

const candidateName = "Gokulnath";

function getCandidateProfile() {}

Use descriptive names.

Prefer:

const activeJobCount = 10;

instead of:

const x = 10;
11. Constants

Use descriptive constants for reusable configuration values.

Example:

const MAX_RESUME_SIZE = 5 * 1024 * 1024;
const DEFAULT_PAGE_SIZE = 20;

Use uppercase naming for global constants where appropriate.

12. React Components

Use functional React components.

Example:

export function JobCard() {
  return <div>Job Card</div>;
}

Keep components focused on a single responsibility.

Avoid creating very large components containing:

UI
API calls
Database queries
Business logic
Validation

all in one file.

13. Component Structure

Recommended component organization:

components/
├── ui/
├── shared/
├── public/
├── candidate/
├── recruiter/
└── admin/

Example:

components/
└── recruiter/
    ├── JobCard.tsx
    ├── JobForm.tsx
    └── ApplicantTable.tsx

Use reusable components where functionality is shared.

14. File Naming

Use consistent file names.

React components:

JobCard.tsx
CandidateProfile.tsx
RecruiterDashboard.tsx

Utilities:

formatDate.ts
validateEmail.ts

API-related files:

route.ts

Avoid inconsistent naming such as:

jobcard.tsx
job_card.tsx
JOBCARD.tsx
15. Tailwind CSS

Use Tailwind CSS for styling.

Prefer utility classes:

<div className="rounded-xl border p-6">

Follow the Career77 Design System for:

Colors
Typography
Spacing
Border Radius
Shadows
Responsive Behavior

Avoid arbitrary styling when an existing design token or utility is available.

16. Tailwind Class Organization

Keep Tailwind classes readable.

Example:

<button
  className="
    rounded-lg
    bg-indigo-600
    px-4
    py-2
    text-white
    hover:bg-indigo-700
  "
>
  Apply
</button>

For complex reusable styling, consider shared components instead of repeating long class lists.

17. Design System

All UI should follow:

docs/07-design/design-system.md
docs/07-design/typography.md
docs/07-design/color-system.md
docs/07-design/spacing.md
docs/07-design/components.md

Do not introduce new colors, fonts, or spacing values without a valid design reason.

18. MongoDB and Mongoose

Use MongoDB as the primary database.

Use Mongoose ODM for:

Schemas
Models
Validation
Database operations

Example:

const JobSchema = new Schema({
  title: {
    type: String,
    required: true,
  },
});

Keep database models organized and reusable.

19. Database Connection

Use a reusable MongoDB connection utility.

Do not create a new database connection for every request.

Recommended:

lib/
└── mongodb.ts

The application should safely reuse database connections where possible.

20. Mongoose Models

Keep models organized.

Example:

models/
├── User.ts
├── Candidate.ts
├── Recruiter.ts
├── Company.ts
├── Job.ts
├── Application.ts
├── Training.ts
└── Subscription.ts

Use clear model names.

21. Database Schema Standards

Schemas should define:

Required fields
Data types
Defaults
Validation
Indexes
Relationships where required

Example:

const JobSchema = new Schema(
  {
    title: {
      type: String,
      required: true,
      trim: true,
    },
  },
  {
    timestamps: true,
  }
);
22. Database Indexes

Create indexes based on actual query requirements.

Examples:

Job Search
Company Lookup
Candidate Lookup
Application Status
Subscription Status

Avoid creating unnecessary indexes.

Indexes should be reviewed as the application grows.

23. Search

At launch, Career77 uses MongoDB text indexes for search.

Use text indexes for relevant searchable fields.

Example:

Job Title
Job Description
Skills
Company Name

Search implementation should be designed so it can later migrate to MongoDB Atlas Search if required.

24. Atlas Search Migration

If Career77 requires:

Better relevance ranking
Advanced search
Larger search scale
More sophisticated search capabilities

the platform may migrate to MongoDB Atlas Search.

The initial search implementation should avoid tightly coupling the application to MongoDB text search behavior.

25. API Routes

Use Next.js App Router Route Handlers.

Example:

app/api/jobs/route.ts

Use appropriate HTTP methods:

GET
POST
PUT
PATCH
DELETE

Use REST conventions consistently.

26. API Structure

API flow:

Request
    ↓
Authentication
    ↓
Authorization
    ↓
Validation
    ↓
Business Logic
    ↓
Database
    ↓
Response

Every protected API must verify authentication and authorization on the server.

27. API Validation

Never trust client-side validation alone.

Validate data on the server.

Validate:

Required fields
Data types
String length
IDs
Email addresses
File information
Permissions

Return clear validation errors.

28. API Response Format

Use a consistent response structure.

Success:

{
  success: true,
  data: result
}

Error:

{
  success: false,
  message: "Unable to process request"
}

The exact API response contract should follow:

docs/08-architecture/api.md
29. Authentication

Use NextAuth.js for authentication.

Authentication should support the Career77 user roles:

Candidate
Recruiter
Super Admin

Authentication logic should be centralized.

Do not duplicate authentication logic across individual pages and APIs.

30. Authorization

Authentication does not automatically grant access.

Always verify permissions.

Example:

Authenticated User
    ↓
Check Role
    ↓
Check Permission
    ↓
Allow / Deny

Example:

Candidate
    → Candidate Portal

Recruiter
    → Recruiter Portal

Super Admin
    → Admin Portal
31. Route Protection

Protected routes should be protected on the server.

Examples:

/candidate/*
/recruiter/*
/admin/*

Do not depend only on hiding UI elements.

Unauthorized users must also be blocked at the server/API level.

32. File Storage

Use Cloudinary for:

Resumes
Certificates
Profile Images

Files should not be stored directly inside the application server.

33. File Upload Standards

Before uploading files:

User Authentication
    ↓
Authorization
    ↓
File Validation
    ↓
File Size Validation
    ↓
Upload to Cloudinary
    ↓
Save Secure File Reference

Validate:

File type
File size
User permission
34. Cloudinary Security

Cloudinary credentials must remain server-side.

Never expose private credentials in client-side code.

Store secrets using environment variables or secure secret management.

35. Image Optimization

Profile images and other supported images should use Next.js image optimization where appropriate.

Use optimized image components and appropriate dimensions.

Avoid unnecessarily loading large original images.

36. Notifications

Career77 uses:

Transactional Email Provider
+
WhatsApp Deep Link

Email should be used for important transactional events.

Examples:

Account Verification
Recruiter Approval
Application Updates
Interview Notifications
Payment Confirmation
Subscription Updates
37. WhatsApp Quick Apply

Career77 may provide a WhatsApp deep link for Quick Apply.

Format:

https://wa.me/<phone-number>

The WhatsApp deep link should be generated safely.

Do not expose private or unauthorized phone numbers.

The Quick Apply experience should follow the Job and Application business rules.

38. Environment Variables

Use environment variables for configuration and secrets.

Examples:

MONGODB_URI
NEXTAUTH_SECRET
CLOUDINARY_CLOUD_NAME
CLOUDINARY_API_KEY
CLOUDINARY_API_SECRET
EMAIL_PROVIDER_API_KEY

Never commit secrets to source control.

Use separate environment configurations for:

Development
Staging
Production
39. Error Handling

Use centralized and consistent error handling.

Do not expose:

Database errors
Stack traces
Secrets
Internal implementation details

to end users.

Log technical details securely on the server.

40. Logging

Logs should help developers identify problems.

Log useful information such as:

Request ID
User ID
API Endpoint
Error Type
Timestamp

Do not log:

Passwords
Authentication Secrets
Payment Credentials
Private Tokens
Sensitive Personal Data
41. Security Standards

All code should follow secure development practices.

Required considerations:

Authentication
Authorization
Input Validation
Rate Limiting
Secure File Upload
CSRF Protection Where Applicable
XSS Prevention
Secure Cookies
Secret Management

Security requirements should follow:

docs/10-development/security.md
42. Code Duplication

Avoid unnecessary duplication.

Instead of repeating:

formatDate()

in multiple files, create a shared utility:

lib/
└── formatDate.ts

Reuse common logic when appropriate.

43. Business Logic

Keep business logic separate from UI components where possible.

Prefer:

UI
    ↓
Action / API
    ↓
Business Logic
    ↓
Database

Avoid putting complex business logic directly inside React components.

44. Comments

Write comments only when they add useful context.

Good:

// Prevent duplicate payment processing for the same transaction.

Avoid unnecessary comments:

// Set name
const name = user.name;

Code should be self-explanatory where possible.

45. TODO Comments

Use TODO comments only for meaningful future work.

Example:

// TODO: Migrate search implementation to Atlas Search
// when relevance ranking requirements increase.

Important TODOs should also be tracked in project issues.

46. Performance

Developers should consider performance during implementation.

Focus on:

Server Components
Efficient Database Queries
Database Indexes
Pagination
ISR
Caching
Image Optimization
Lazy Loading
Minimal Client JavaScript

Avoid unnecessary API calls and database queries.

47. Pagination

Large datasets must use pagination.

Examples:

Jobs
Applicants
Applications
Users
Transactions
Support Tickets

Prefer server-side pagination for large datasets.

48. Forms

Forms should provide:

Clear labels
Validation
Error messages
Loading states
Success feedback
Disabled submit state during processing

Example:

Submit
    ↓
Validate
    ↓
Submitting...
    ↓
Success / Error
49. Accessibility

All components should follow accessibility standards.

Ensure:

Keyboard navigation
Proper labels
Focus states
Semantic HTML
Accessible forms
Sufficient color contrast
Screen reader support
50. Testing

Code should be tested according to feature complexity.

Test:

Components
API Routes
Business Logic
Authentication
Authorization
Database Operations
Payment Flows
File Uploads

Critical features require stronger test coverage.

51. Git Standards

Use meaningful commit messages.

Examples:

feat: add recruiter job creation
fix: resolve application status issue
refactor: simplify candidate profile service
docs: update recruiter workflow
chore: update dependencies

Avoid vague commits:

update
changes
fix
test
52. Pull Request Standards

Every Pull Request should clearly describe:

What changed
Why it changed
How it was tested
Screenshots where applicable
Known issues

Before creating a Pull Request:

Run Lint
Run Type Check
Run Tests
Run Build
